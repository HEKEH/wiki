# 新架构线程模型

理解 RN 性能与卡顿的根本视角:哪些工作在哪个线程,以及为何 60fps 需要两个线程都不掉帧。

## 两个主要线程

| 线程 | 职责 |
|------|------|
| **JS 线程** | React render 阶段、业务逻辑、API 调用、触摸事件处理、state 管理;并**驱动**布局计算(调用原生 Yoga 引擎,见下)。渲染管线最常见路径。 |
| **UI(主)线程** | **唯一能操作原生视图的线程**;高优先级事件处理;原生动画、ScrollView 滚动(独立于 JS 线程);特定场景下同步执行整条渲染管线。 |

> "两个主要线程"是官方新架构 threading-model 的说法,指日常盯性能的两条关键线程(两个 frame rate)。**但 RN 实际不止两条** —— 布局有后台线程、原生模块有各自的线程,详见下文 [完整线程清单](#完整线程清单)。

## 完整线程清单

| 线程 | 跑什么代码 | 职责 |
|------|-----------|------|
| **JS 线程** | JavaScript(Hermes 引擎内) | React 协调/render、业务逻辑、API、事件分发。**你的 JS 全在这一条上跑**(JS 单线程) |
| **UI / 主线程** | 原生(ObjC/Swift、Java/Kotlin) | 唯一能操作原生视图;触摸手势、原生动画、滚动、落地 mutation |
| **后台线程 / Shadow 线程** | 原生 C++ | 布局计算(Yoga);旧架构里专门叫 "Shadow 线程" |
| **原生模块线程** | 原生 | 各 Native/TurboModule 可有自己的队列或线程池,做网络/文件/DB 等异步活 |

**"后台线程"是个泛称** —— 指既不是 JS 线程、也不是 UI 主线程的工作线程,目的就是把耗时/可并行的活从两条关键线程挪走(那两条各有"绝不能卡"的理由:UI 卡→掉帧,JS 卡→触摸/逻辑冻结)。

**后台线程 vs JS 线程**:并列的独立线程,各跑各的、语言不同(JS 线程跑 JavaScript,后台线程跑原生 C++ 如 Yoga)。它们是**接力协作**,对应渲染管线三阶段(见 [[concepts/02-fabric-render-pipeline]]):

```text
Render(JS 线程,建 Shadow Tree)
   → Commit / 布局计算(JS 或后台线程,跑 Yoga)
      → Mount(UI 主线程,落到真实视图)
```

> **Yoga 说明**:Yoga 是 Meta 开源的、用 **C/C++** 写的 Flexbox 布局引擎,**不是 JS**。你用 JS 写样式,但布局计算由原生 Yoga 引擎完成。它可在 **JS 线程或后台线程**执行(取决于场景),不阻塞 UI 主线程 —— 所以本页把它列在"驱动方"而非"实现语言"角度。

### 新旧架构差异

- **旧架构**:有明确独立的 Shadow 线程算布局;JS↔原生经 Bridge,Bridge 自带消息队列线程。
- **新架构**:JSI 去掉了专门的 Bridge 线程(C++ 直接调用);布局线程更灵活(JS 或后台),旧 Shadow 线程的活被整合进管线,故官方只强调"两条主线程"。

## 帧预算:两个 frame rate

目标 60fps → 每帧 **16.67ms**。

- **JS frame rate**:JS 线程若一帧内无响应即丢帧。例:复杂组件 200ms 重渲染 ≈ 丢 12 帧,JS 动画与触摸冻结。更新在每次事件循环末尾批量发往原生。
- **UI frame rate**:原生动画与滚动跑在主线程,对 JS 卡顿更有韧性。

→ 诊断卡顿先分清是 JS 线程还是 UI 线程掉帧(Perf Monitor 分别显示)。详见 [[concepts/04-performance-optimization]]。

## 线程安全靠不可变性

高层次上,线程安全由**不可变数据结构**保证(C++ const correctness):每次 React 更新在渲染器内 clone/新建对象而非原地修改。因此框架可暴露**线程安全的同步 API**,无需传统锁。

## 渲染管线的五种线程场景

前提:React 18 的渲染**分小块进行、会周期性让出**,所以"渲染到一半被打断"才成为可能。哪种场景取决于**触发更新的事件优先级**(见下方 [事件优先级](#事件优先级)):

1. **标准**:渲染管线主要在 JS 线程异步执行(最常见路径)。
2. **UI 线程渲染**:高优先级事件触发整条管线**在 UI 线程同步执行**,保证交互当场响应。
3. **低优先级中断**:JS 线程正在 render 时来了**低优先级** UI 事件——不紧急、不抢占,render **继续留在 JS 线程**,事件排队等待。
4. **高优先级中断**:render 途中来了**高优先级**事件(如点击)——当前 render 被打断,**改到 UI 线程同步完成**,让交互立刻拿到最新 UI。
5. **C++ state 更新**:源自 UI 线程(如滚动),直接**跳过 render 阶段**(见 [[concepts/02-fabric-render-pipeline]] 的两条状态更新路径)。

> 第 4 条"改到 UI 线程同步完成":官方 threading-model 是高层描述,**未言明底层是"从断点恢复"还是"以高优先级重启渲染"**;就 React 并发模型的通用行为,高优先级抢占常是丢弃进行中的低优先级工作、在高优先级下重跑。此处只确定"结果是转到 UI 线程同步完成"。

**几个易误解的点(第 4 条):**

- **"同步"= synchronous(阻塞、一口气跑完),不是"把数据 sync 给 UI 线程"。**
- **"渲染"= React 的 render 阶段**(执行组件、协调、产出完整的 Element/Shadow 树),**不是画像素**;画到屏幕在其后的 Mount。
- **中断 ≠ 目标达成**:屏幕此刻还没显示该显示的新状态,这笔账仍欠着,所以**必须**渲染;渲染的是**当前待处理的更新**(通常即高优先级事件带来的新状态)。
- **旧半成品常已作废**:高优先级更新往往改掉了渲染所依赖的状态,进行到一半的旧渲染基于过时状态、结果已错,只能丢弃并用最新状态重渲染。
- **Mount 只针对完整树**:React 绝不挂载半成品,否则界面会出现"一部分更新、一部分没更新"的撕裂。
- 例:输入框已输入 `ab`(低优先级过滤大列表进行中),再敲 `c` → 高优先级更新须**立刻**把输入框渲染成 `abc`;那半成品"过滤 `ab`"因查询已变而作废,稍后再低优先级跑"过滤 `abc`"。若不重渲染,输入框就会卡在错误的 `ab`。

### 事件优先级

React/Fabric 按"用户是否期待即时反馈"给更新分优先级:

| | 高优先级(离散 discrete) | 低优先级(连续 continuous + 过渡) |
|---|---|---|
| 典型 | 点击、按下、文本输入 | 滚动、指针移动;`startTransition` 标记的更新、数据加载 |
| 期待 | **必须立刻响应** | 可延后、可被打断 |

- 开发者可用 `useTransition` / `startTransition` **主动把某次更新降级为低优先级**(可被紧急交互抢占),呼应 [[concepts/01-new-architecture]] 的并发特性。
