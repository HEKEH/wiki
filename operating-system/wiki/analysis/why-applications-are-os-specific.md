# 为什么应用程序与操作系统绑定（综合）

常识认为程序跑不了是因为 **CPU 架构不同**(ARM vs x86)。但这只是一半:**应用程序不仅与架构绑定，
也与操作系统绑定**。两个反例证明问题不在 CPU：

- 2019 年的 Intel Mac 与 Windows 同用 x86，Windows 程序在 Mac 上仍跑不了。
- 同一台机器双系统(Linux + Windows，同一 CPU)，Linux 上照样跑不了 Windows 程序。

既然机器码就是 CPU 能执行的指令，同 CPU 上为何不通用?因为"能被 CPU 执行"≠"能被操作系统加载并
正确服务"。下面四层约定**任何一层不对齐**都会导致失败。

## 第一层：系统调用集合不同

用户程序跑在用户态、不能碰硬件，凡涉及硬件的事都要经 [[concepts/system-calls]] 请操作系统代劳
(背景见 [[concepts/mode-bit-and-operational-modes]])。程序对系统调用依赖极深(Linux 上看
`/proc/<pid>/status` 的 `voluntary_ctxt_switches`，数字巨大)。而**每个操作系统提供自己独有的
一套系统调用**——这是不兼容的第一来源。

**不止命名差异，语义也不同**。经典例子是进程创建：

| | Windows `CreateProcess` | Unix `fork` (+ `exec`) |
|---|---|---|
| 做什么 | 接收可执行文件路径，直接新建进程运行它 | `fork` 创建**调用进程的克隆**(仅 PID 不同) |
| 运行特定程序 | 一次调用即可 | `fork` 后子进程再调 `exec`，用目标文件内容**替换自己的 text 段** |

于是同一件事：一个平台一次系统调用、另一个要好几次。**编译成机器码后，即便同一 CPU 架构，两边也
生成不同指令**；多出来的指令在另一个 OS 上毫无意义 → 未定义行为/崩溃。(`fork`/`exec` 本身是个
大话题，值得单独展开。)

## 第二层：ABI 不同

即便两系统定义了**完全相同**的系统调用集合，更底层的二进制约定仍可能不一致——这套约定叫
[[concepts/application-binary-interface]]。关键对齐点(以系统调用为例，详见 [[concepts/syscall-abi]])：

- **调用号放哪个寄存器**：一方换个寄存器，另一方就会读错、认错系统调用。
- **参数怎么传**：寄存器 / 内存 / 混用因 OS 而异(Linux：≤6 个放寄存器，超出写内存)。
- 参数在内存时**用栈还是专门位置**；含**指针**时**地址宽度**(32/64 位)也须一致。

这些"看似微小"的细节任何一处错位，程序都可能在另一 OS 上**勉强执行却行为不可预测**。

## 第三层：可执行文件格式不同

可执行文件**不只含指令**，还含**数据 + 元数据**，告诉操作系统程序各部分位于何处、如何加载执行
(并支持链接/调试/动态库)。**每个 OS 有自己的格式规则**——见 [[concepts/executable-file-format]]：

- **ELF**(Linux) vs **PE**(Windows)。

操作系统按自己认识的格式解析文件头；格式不对，连"把程序装进内存"这一步都过不了。

## 第四层：运行时依赖

有些语言不编译成机器码，而跑在**虚拟机/解释器**上——装了对应运行时，同一份代码即可跨平台。**这
本是为绕开上述兼容性问题而设计的**。但现代应用常**模块化**：只要有一个模块依赖的运行时未安装，
整个应用就可能失败。所以运行时既是跨平台的解药，也可能是新的依赖陷阱。

## 结论

> "能在同一 CPU 上执行"只是必要条件。一个程序要在某 OS 上**真正运行**，还须：调用的系统调用在
> 该 OS 存在且语义一致、二进制层 ABI 对齐、可执行文件格式被该 OS 识别、所有依赖的运行时已安装。
> 任何一层不满足即失败——这就是应用程序**与操作系统绑定**的根本原因。
