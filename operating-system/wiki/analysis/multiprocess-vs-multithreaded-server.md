# 多进程 vs 多线程服务器

以一个"返回客户端头像"的服务器为例(监听 3000 端口)，对比用**多进程**还是**多线程**来消除
[[concepts/concurrency]] 中的**阻塞效应**。

## 问题:顺序处理导致阻塞

单线程服务器顺序执行三步:

```
accept(接受) → process(从磁盘查找并读取图片，昂贵的 IO) → respond(发回)
```

`process` 的磁盘 IO 远比 accept/respond 耗时。多个请求几乎同时到达时，第二个必须等第一个
**完整处理完**才被受理。一千个请求时，时间线出现大片 CPU **闲置空隙**——本可用于受理并开始
处理其他请求的时间被白白浪费。这就是**阻塞效应**。

## 方案 A:每客户端一个进程

一个 **Listener 进程**监听;每来一个请求就**新建一个进程**去服务该客户端，受理与服务并发进行。
缺点:

- **内存低效**——每个进程有独立地址空间，扩到数千客户端时开销巨大。
- **创建开销大**——派生进程非平凡操作，吃掉本想省下的 CPU 时间。
- **全局状态难维护**——子进程间共享状态需 [[concepts/interprocess-communication]] 同步，实现复杂。

## 方案 B:每客户端一个线程

为每个到来的客户端**派生一个 [[concepts/threads]]**，效果类似方案 A，但:

- **内存省得多**——所有线程共享同一进程的地址空间。
- **创建更快**——创建线程的系统调用通常远快于创建进程。
- **共享状态更简单**——线程默认共享内存(但仍需同步以避免竞态)。
- 所有 handler 线程**指向内存中的同一个处理函数**(线程不含代码，只用 PC 指向 text 段;
  并发读同一段代码是安全的)。

## 对比小结

| 维度 | 多进程 | 多线程 |
|---|---|---|
| 地址空间 | 各自独立 | 共享(同一进程) |
| 内存开销 | 高 | 低 |
| 创建速度 | 慢 | 快 |
| 共享/全局状态 | 需 IPC | 默认共享(需同步) |
| 隔离性/容错 | 强(一个崩不影响其他) | 弱(共享地址空间) |

> 取舍:多线程在内存与创建速度上明显占优，是高并发服务器的主流选择;但因共享地址空间，**容错与
> 隔离不如多进程**(参考 [[concepts/shared-memory-ipc]] 里 Chromium 反而用多进程换隔离的例子)。
