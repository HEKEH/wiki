# 消息传递（IPC）

**消息传递**让进程在**不共享地址空间**的前提下通信与同步，是
[[concepts/interprocess-communication]] 的两种基本模型之一。地址空间保持隔离，进程通过发送
消息交换数据。常见机制：**管道（pipes）**、**套接字（sockets）**、**远程过程调用（RPC）**。

## 机制：内核中的邮箱

设进程 A 想给进程 B 发消息：

1. A 发起系统调用，请操作系统与 B 建立**链接（link）**——一条逻辑路径。
2. 内核实际在**自己的地址空间**里创建一个**队列（queue）**作为**邮箱（mailbox）**，A 发、B 收。
3. 队列行为可变：同步 / **异步**、**带缓冲**（限制消息数）。**全双工**可用两个队列实现，使双方
   能同时互发而不冲突。

由于邮箱在内核地址空间，进程不能直接访问，操作系统必须提供至少两个系统调用：

- **send**——A "无法直接访问邮箱，请帮我把消息放进去"；操作系统把消息**复制**进邮箱。
- **receive**——B 请操作系统检查邮箱；若有消息，作为函数返回值返回。

（为何必须用系统调用，见 [[concepts/system-calls]] 与 [[concepts/mode-bit-and-operational-modes]]。）

## Mach 与端口

最早引入共享邮箱概念的系统之一是 **Mach OS**，其衍生系统是 iOS / macOS 的基础。Mach 把进程
当作**任务（task）**，把邮箱称为**端口（port）**。要点：

- 消息发往**端口**而非直接发往进程——两个进程可有多条链接，且并非所有端口都恰好关联两个进程。
- **监听端口（listening port）**——进程保留的开放端口，接收任意进程的**连接请求**，用于建立
  新的私有通信链接。

## 术语辨析：邮箱 / 端口

> 以下"同义"层来自源材料，"Mach 权限"与"套接字端口"两层为外部工程知识，用于消除本页"端口"
> 一词的歧义。

- **同义层**：在本页与经典教材里，**邮箱(mailbox)** 是这个抽象的通用叫法(内核里的命名消息
  队列)，**端口(port)** 是 Mach 给它起的专有名字——同一概念的两个名字。视频强调的"消息发往
  端口而非进程"正是这一抽象的核心。
- **Mach 端口 ≠ 朴素邮箱**：Mach 端口还带一层**端口权限(port rights)** 能力模型——**接收权**
  同一时刻只能由一个任务持有(唯一收件人)，**发送权**可由多个任务持有(多个寄件人)。这解释了
  "并非所有端口都恰好关联两个进程"以及监听端口的存在。朴素邮箱模型通常不强调这套权限语义。
- **套接字的"端口"是另一回事**：上一节套接字里的端口是一个 **16 位编号(0–65535)**，用于在
  一台机器上**复用/标识**服务(80=HTTP、22=SSH)，是"找到正确邮箱的地址编号"，而**不是**队列
  对象本身。它与 Mach 端口概念类似(都是进程接收消息的入口)，但技术上是不同的东西。
  - 即便在 macOS 上两者也无一一对应：它们分属 XNU 的不同层——**Mach 端口**在 Mach IPC 子系统，
    **套接字端口**在 BSD 网络栈，各走各的接口(`mach_msg` vs `socket`/`bind`)。命名相同纯属概念
    类比，底层是两个无关对象。

## 跨机器通信与套接字

最大优势：进程不必共享地址空间，因此**不必在同一台机器**上。底层需网络能力与网卡驱动级编程，
但若在操作系统层面正确实现，对开发者无缝——跨机器与同机器用法相同。

**客户端—服务器**架构通常用**套接字接口**实现（几乎所有主流操作系统都支持）。客户端与服务器
其实是**进程**而非机器；通信不一定需要网络（如 localhost 测试）。请求中 **IP 地址**标识机器，
**端口**代表服务器进程接收请求的邮箱。HTTP、FTP、SSH 等服务都依赖套接字接口。

## 缺点

端口在内核地址空间，**每次收发都要走系统调用**，性能开销较大——这正是
[[concepts/shared-memory-ipc]] 没有的限制。但并不是说消息传递慢，只是共享内存更快；99% 的
场景下消息传递已足够。
