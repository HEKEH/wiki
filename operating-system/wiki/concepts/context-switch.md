# 上下文切换（Context Switch）

**上下文切换**指：当一个进程被中断、要把 CPU 让给另一个进程时，操作系统**保存被中断进程的
寄存器状态、再载入下一个进程的状态**的过程。"上下文"即 CPU 运行所依赖的寄存器等状态。

## 为什么必须做

一条指令的结果取决于执行时的**整个上下文**，尤其是**寄存器的值**。中断发生时，寄存器里的数据
属于**被中断的进程**；若直接把 CPU 交给新进程，它就会在**错误的上下文、用错误的数据**运行。所以
必须先把旧状态存入内存(最终落在该进程的 **PCB / `task`**，见 [[concepts/threads]])、再载入新状态。

> 上下文切换复用 [[concepts/interrupts]] 处理例程的"保存/恢复状态"机制，但**特指恢复时载入的是
> 另一个进程**的情形(若恢复同一进程只是普通中断返回)。它**不会自动发生**——需要操作系统**真正执行
> 代码**(并借助硬件，见下)。

## 核心难题：纯软件做不到

[[concepts/interrupts]] 的两个效果之一是：**PC 被立即覆写**以跳转到处理程序。问题在于——**早在取
第一条处理指令之前，PC 就已被覆写**，进程"在哪里被中断"的信息已丢失。没有任何软件指令能抢在最
开头保存 PC。此外，处理代码由操作系统执行，而**操作系统也有自己的栈**，但中断时**栈指针仍指向
用户栈**，贸然压栈会破坏被中断进程的栈。

> 结论：保存被中断进程状态(尤其是 PC)**无法仅靠软件完成**。这是一个**罕见的"硬件必须支持软件"**
> 的情形——架构至少要提供保住 PC 的最低硬件支持。

### 为何协作式切换不受此难题困扰

PC 丢失是**异步中断(抢占)独有**的问题：硬件在任何软件指令执行前就覆写 PC。而**协作式**切换是程序
**自愿、同步**地让出——它执行一条指令去切换(一次普通函数 `call`，或自己发起的软件陷阱)：

- **PC 由调用约定保住，而非被异步摧毁**：`call` 会同步保存返回地址(切换点的 PC，x86 压栈、ARM/RISC-V
  存链接寄存器)，`ret` 再恢复。PC 在程序**自选**的位置被规矩存下，没有"还没来得及跑就被清掉"的窗口。
  > 注意：`call` 当然也要改 PC，但"保存旧 PC"是它**指令自身**的原子行为——硬件在执行这条指令、跳转
  > **之前**就把旧 PC 存好(此刻旧 PC 仍在 CPU 内部)。真正的差别不是"call vs 中断"，而是**硬件在转移
  > 过程中保存(拿得到旧 PC) vs 软件在转移之后才动手(旧 PC 已没)**。因此下文的"硬件支持"本质上就是
  > **让中断像 `call` 一样、由硬件在跳转前自动存好 PC/SP**(x86 中断自动压 `CS:EIP` 即是一例)。
- **SP/寄存器可提前保存**：切换时机由程序掌控，yield 例程在交出控制权**前**就用普通 store 指令把 SP、
  寄存器存进自己的上下文。
- 在没有中断机制的极简 CPU 上压根没有异步中断，此难题**自始不存在**。

> 附带差别：抢占式中断点是**任意指令**，必须保存**整套 CPU 状态**；协作式切换点固定在**函数调用
> 边界**，受 ABI 约定管辖，通常只需保存 **callee-saved 寄存器 + SP**，更轻。代价是**依赖程序诚实让出**
> ——死循环不 yield 就会霸占 CPU，这正是协作式被淘汰、改用定时器抢占的原因
> (见 [[concepts/preemptive-vs-cooperative-os]])。这类极简 CPU 罕见，既不安全也不实用。

## 两类硬件支持

### ① 多寄存器组（multiple register sets）

CPU 提供多组寄存器分给不同进程，被中断时换用另一组，从而不动前一进程状态。无需无限多组——
**两组即可**，配合操作模式：

- **普通组**：用户程序共享；**特权组**：仅操作系统在内核态可访问。
- 中断时 CPU 切内核态、停用普通组、启用特权组 → 中断**不会覆写普通组上的 PC**。
- 操作系统用**特权指令**读被停用组的值、经普通指令写入内存(最终存进被中断进程的 PCB)；载入新进程
  时做相反操作；若抢占式则用特权指令设定时器；最后一条**中断返回特权指令**切换活动寄存器组并切回
  用户态，恢复被选中的进程。

> 现代架构**少用**此法：复制整套寄存器电路不划算(何况现代有数百寄存器乃至 SIMD 寄存器)，且其实
> 只需为 **SP、PC 等关键寄存器**单独留一份即可。

### ② 硬件自动化部分步骤

把若干步骤**硬连线进 CPU**，在中断生效前自动执行：

- **自动保存关键寄存器**：让操作系统栈指针始终对 CPU 可用(硬连线内存位置或专用寄存器)；中断时在
  PC 被覆写前，硬件**自动把 PC、SP 压入操作系统栈**，其余寄存器再交软件保存。
- **几乎全硬件切换**：用一个**段寄存器**指向操作系统管理的内存，其中的特殊数据结构在某些架构称
  **任务状态段(TSS)**、对应**任务寄存器(task register)**；中断时 CPU **自动把所有寄存器复制进 TSS**。

> 全硬件方案(TSS)**确实可让切换从软件视角"自动发生"，但主流操作系统大多避免使用它**。
