# 并行（Parallelism）

**并行**指系统**真正同时**执行多个任务的能力。它需要**多核**(或多处理器)硬件：系统把不同
[[concepts/threads]] 分派到不同核心上同时运行。这与 [[concepts/concurrency]] 不同——并发只要求
所有任务都能取得进展，单核靠快速交错即可实现。

## 并发 vs 并行

| | 并发（Concurrency） | 并行（Parallelism） |
|---|---|---|
| 定义 | 支持多任务，让所有任务都能取得**进展** | 让多任务**真正同时**进行 |
| 硬件 | 单核即可 | 需要多核/多处理器 |
| 机制 | 快速**交错(interleaving)** CPU 访问，制造"同时"的错觉 | 每个线程占一个独立核心 |

> 关键：**可以有并发而没有并行**(单核就是如此)；反之并行一定也是并发。并行的优势在于多任务的
> 流畅度不再那么依赖"快速交错"制造的错觉。

## 为何需要更多核心

并发有极限：并发任务太多时，即使 CPU 轮换再快，每个任务**重获 CPU 的间隔也会变得太长**，系统
不再流畅。三种解法：①让 CPU 更快(过去十年越来越难，且任务继续增长又回到原点)；②更聪明地调度
CPU 访问(更复杂，源材料留作单独一集；可参考 [[concepts/preemptive-vs-cooperative-os]] 的调度基础)；
③**加更多处理器**——多核正是这条蛮力路线。

**术语**：CPU 常指整个封装/芯片；封装内**每个核心(core)** 是独立处理单元(与其他核心共享缓存等
部件)。无论如何，**每个核心在操作系统看来都是一个独立的处理器**。

## "最多 n 个"线程并行

若系统有 **n 个核心**，任意时刻**最多有 n 个线程并行**——注意是"最多"：

- **核心数固定**：创建一千个线程不代表一千个任务真的并行。
- **线程要竞争资源**：即便线程数正好等于核心数，**其他进程的线程**也要 CPU 时间；操作系统为
  公平分配 CPU，会限制我们自己能并行的线程数。

## 并行的单位：进程也能并行吗？

**能。** 并行的单位是**任何可被调度的执行实体**，不是"线程"这个特定概念。操作系统调度器把任何
就绪的执行单位派发到任何空闲核心——在"进程=最基本执行单位"的旧模型里，并行就发生在**进程之间**，
这在多处理器系统里一直是常态，且早于多线程的普及。上一集"每客户端一进程"的服务器
([[analysis/multiprocess-vs-multithreaded-server]])在多核上已是真正的并行。Linux 里进程与线程都是
同一个 `task`、同一个调度器派发，"谁能并行"对两者一视同仁(见 [[concepts/threads]])。

所以真正的区别是**进程间并行 vs 进程内并行**：

- **线程的独到之处不是"能并行"，而是"让同一进程内部也能并行"**——线程出现前，单进程只有一个
  程序计数器，一次最多用一个核；有了线程，一个进程才能**横跨多个核**。
- **成本**：进程间并行要为每个单位付出独立地址空间(内存高)、创建慢、共享状态还得走
  [[concepts/interprocess-communication]]；线程共享地址空间，更轻更省、创建更快——代价是同步负担。

因此实践中要在**应用内**利用多核，线程是默认手段。

## 可移植性：声明并发即可

程序员要让任务并行，只需把它们**声明为用线程并发**，其余交给操作系统：无空闲核心就交错执行，
有多核就每线程分一个核心。因此程序更**可移植**——不必为特定核心数编译。这也解释了为何现代
操作系统几乎都以 [[concepts/threads]](而非进程)作为最基本的执行单位。

## 两类并行

- **数据并行(data parallelism)**：把**同一份数据**的子集分到多个核心，每核对自己的子集执行
  **相同操作**。例：在大数组里找质数——每个数是否为质数互不依赖，可均分成四份各给一核。
- **任务并行(task parallelism)**：分布的是**任务/线程**，每个线程执行**不同操作**；这些线程可能
  操作**同一份数据**，也可能操作**不同数据**。例：对同一数组同时求最小值/最大值/均值/是否含 101——
  每个操作都需访问整个数据集，故不切数据而把每个操作分给一核。

> **加速比 < 核心数**：把工作分到 n 核**不一定**带来 n 倍性能(同步、数据划分、共享部件等都有
> 开销)。但相比单核顺序执行仍是显著提升。
