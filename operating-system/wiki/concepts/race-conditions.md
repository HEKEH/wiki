# 竞态条件（Race Conditions）

**竞态条件**是一种软件缺陷：程序的结果取决于多个线程/进程访问与修改**共享资源**的**不可预测的
先后顺序**。多个操作"赛跑"去读写同一份数据，谁先到终点决定了最终结果——故名"竞态"。它通常是
**抢占式操作系统**（见 [[concepts/preemptive-vs-cooperative-os]]）的副作用：那个"不可预测"正源于
抢占的随机时机。

## 根因一：大多数操作不是原子的

理解竞态条件的第一关键概念：**一行源代码 ≠ 一条 CPU 指令**，多数操作需要多个 CPU 步骤，因而**不
是原子的**——执行到一半可能被打断。

- **加法 `a + b`**：把**每个**操作数从内存取入寄存器（两条）→ 令 CPU 相加 → 把结果写回内存，共四步。
- **拷贝字符串**：C 字符串是以 null 结尾的字符序列；拷贝要**逐字符循环**直到 null（即便按块拷贝，
  块也有最大尺寸，长串仍需多步）。
- **自增 `count++`**：读取 → +1 → 写回，是典型的**读-改-写**三步。
- **连读取并比较一个标志位也不原子**：载入内存值到寄存器 → 与期望值比较 → 据结果行动。

只要一个操作横跨多步，中途被切走，共享数据就可能停在一个**我们从未打算让它出现的中间状态**——
此即**无效状态(invalid state)**。

## 根因二：抢占带来的不可预测时机

抢占式系统会周期性强行打断进程，让别的进程用 CPU（见 [[concepts/preemptive-vs-cooperative-os]]、
[[concepts/cpu-scheduling]]）：

- **时间片(time quantum)** 到期 → 硬件定时器触发 [[concepts/interrupts]] → 进程被抢占。
- 若一次修改**耗时超过一个时间片**，则**至少保证被打断一次**，数据必然进入无效状态。
- 但**更短的操作也不安全**：①操作恰好起于时间片末尾，极易被打断；②中断不止来自调度器定时器——
  UI 按键、网络数据包、其他进程设的定时器都会触发中断，**无论来源都能抢占当前线程**，哪怕时间片
  没用完。这些事件无法阻止（否则系统就对它们失去响应）。

## 两类典型场景

### 1. 单写多读：脏读到中间状态

源材料的服务器例子：一个 writer 线程周期性更新共享字符串，多个 handler 线程读取它回应客户端。
writer 拷贝字符串拷到一半被抢占，handler 此时读取，便拷走了**半新半旧、拼接自不同句子的乱串**。
极其罕见——9 百万请求里出现 1 次、8500 万请求里 14 次，几乎只能靠专门写程序扫异常才能发现。

### 2. 多写：丢失更新(lost update)

多个线程同时对**共享计数器**自增（如并行统计质数个数）。由于自增非原子：

- **并行下**：两线程同时读到同值，各自 +1 再写回，两个质数却只让计数 +1（无需任何中断，纯并行
  访问所致——见 [[concepts/parallelism]]）。
- **抢占下**：线程 A 读值 10、+1 得 11，写回前被抢占；其余四线程顺利把计数推到 14；A 恢复后用陈旧
  的 11 **覆写**，计数从 14 跌回 11——四个正确结果凭空丢失。

> **非确定性**：同一程序多次运行结果几乎每次不同，有时甚至"碰巧正确"。这正是竞态条件的标志，
> 也是它最危险之处——**极其罕见、难以复现与调试**。轻则崩溃，重则行为完全不可预料。

## 为何不能用普通变量当锁

直觉做法是加一个**控制变量/标志**（1=正在写，0=空闲），读者先查标志再决定是否读。但**标志本身
也是共享数据，同样会竞态**：读者载入标志值到寄存器、刚比较完就被抢占；writer 随后把标志置 1 并
开始修改、也被抢占；读者恢复时其寄存器里仍是**过期的旧标志值**，于是误判"安全"而读到无效数据。
症结：读-比较-行动并非原子。

像 **Peterson 算法**这类纯软件方案能在很特定的条件下工作，但依赖大量微妙假设（软件行为、编译器
优化、硬件架构），脆弱不可靠。健壮的解法需要**硬件层面的同步原语**（见 [[concepts/concurrency]]
末节）——那是线程同步的话题，留待后续。
