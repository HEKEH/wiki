# 线程（Thread）

**线程**是进程内部、可被操作系统独立调度的**最基本执行单位**。它的出现是为了在**单个进程内**
实现 [[concepts/concurrency]]，从而解决进程级方案的**阻塞**与开销问题。

## 为什么需要线程

传统上进程是最基本的执行单位，每个进程只有**一个程序计数器(PC)**，因此同进程内的两段代码
(如两个函数)无法并发——单个 PC 记不住两处各自的中断位置。把代码拆成多进程 + IPC 虽可行，但
不直观，且**每客户端一进程**的服务器方案在数千并发时**内存低效、创建开销大、全局状态需 IPC
同步**(对比见 [[analysis/multiprocess-vs-multithreaded-server]])。

解决办法:**不再把 PC 绑定到进程**，而是给进程内每个要并发的内部实体各分配一个 PC——这就是线程。

## 共享与独占

线程属于某个进程，**共享该进程的整个地址空间**(text/data/堆、打开的文件、IO 设备)，但因为会被
中断让出 CPU，必须能保存/恢复各自状态，所以每个线程**独占一份 CPU 状态**:

| 独占（每线程一份） | 共享（整个进程） |
|---|---|
| 程序计数器 PC | 地址空间（代码段、堆） |
| 寄存器组、标志位、累加器 | 打开的文件 |
| 栈指针 + 独立的**栈** | IO 设备等资源 |

- **各自的栈**——否则并发函数共用一个栈会互相覆盖局部变量。
- 但"各自的栈"≠"不能访问彼此的栈":地址空间是**进程**的属性，栈就在这块共享空间里，一个线程
  物理上**能**读写另一个线程的栈——只是一般**不应**这么做。
- 线程间要用共享内存通信，**堆**比栈更合适;但读写同一块内存仍需**同步**(见 [[concepts/concurrency]]
  的竞态/阻塞讨论)。
- 不同进程的地址空间**仍隔离**:同进程兄弟线程默认共享内存，**跨进程线程**共享数据仍需
  [[concepts/interprocess-communication]]。

## 线程不是什么

- **不是函数**:线程执行的代码在 **text 段**;线程不含代码，只用 PC **指向**代码。因此多个线程
  可指向**完全相同的代码**(服务器例子中所有 handler 线程指向同一函数)。
- 并发访问同一段代码**安全**:text/data 段**从不被写入**;线程运行时的专属数据都在**栈或堆**。
- 这解释了 C 中**创建线程要传函数指针**——传的是线程代码的起始地址。
- **不是异步函数**。

## 实现要点

- **PCB 与 task**:用多线程取代多进程需重写进程控制块(PCB)。Linux 内核用**同一个结构 `task`**
  表示进程和线程，抽象掉两者区别——并发统一描述为"轮换 task"，用同一个调度器调度。
- **主线程(main thread)**:每个进程至少有一个，创建进程时自动生成;即使不用多线程，OS 也靠
  调度主线程来分配 CPU。很多实现里主线程标识进程——主线程结束则其余线程**立即终止**。
- **动态创建**:多数主流 OS 让进程以单线程启动、运行时按需经系统调用创建线程(编译期难以预知
  线程数)。创建线程的系统调用通常**比创建进程快得多**,内存也省得多。

## 两种定义

- **OS 视角**:最基本的执行单位(即上一代模型里"进程"的角色)。
- **开发者视角**:告诉 OS"这些代码片段可并发执行"的机制。
- 亦称**轻量级进程**——更易、更快创建。

> 本页内容在**单核**处理器上同样成立(线程在单核上用于填补 IO 空隙)。多核上的额外用途是
> **并行(parallelism)**，属另一主题。
