# 内存保护（基址与界限寄存器）

**核心规则：默认情况下，一个进程永远不能访问另一个进程的地址空间。** 本页讲操作系统如何在硬件
协助下强制这条规则——这是历史上最简单的硬件内存保护方案：**基址寄存器 + 界限寄存器**。

## 地址空间与为何必须隔离

程序在磁盘上，但 CPU 只有 `load`/`store` 这类**以内存地址为参数**的指令，没有任何指令吃磁盘
地址；所以程序必须先被**载入内存**才能被取指执行，成为**进程**。进程为变量、临时值等还要额外
内存——分给一个进程的全部内存就是它的**地址空间(address space)**。

为何地址空间"神圣不可侵犯"？源材料的支付表单例子：

- **被读**：进程 A 的表单把用户输入存在缓冲区里；若无保护，进程 B 可用 `load`/`store` 读走 A 的
  支付数据、拷进自己空间、经网络发给攻击者。（我们安装来路不明的 app 时尤其危险。）
- **被写**：B 还能改写 A 的缓冲区，篡改支付金额或收款人而用户无察觉。
- **改代码**：更糟的是 B 能改 A 甚至**操作系统自身**的代码，注入恶意指令为所欲为。

操作系统自己也靠内存管理系统，所以**无保护的内存访问是任何现代 OS 都不能容忍的**。

## 为何纯软件做不到

进程运行时内存操作**无时无刻不在发生**（每条指令都要先从内存取指=一次读，`load`=一次读，
`store`=一次写——连自增计数器都涉及多次访存）。若让 OS 对每个进程的**每一次**访存都跑额外代码
去校验，等于为用户的每条指令再多跑好几条指令，**开销巨大**。所以**必须有硬件支持**（与
[[concepts/context-switch]] 同属"硬件支持软件"的情形）。

## 硬件方案：基址 + 界限

内存本身不动——内存单元只通过**地址总线(address bus)** 看到一串地址，并不知道这些地址是怎么
产生、作何用途。保护逻辑放在 CPU 里，用两个**特权寄存器**：

- **基址寄存器(base register)**：进程**最小的合法物理地址**。
- **界限寄存器(limit register)**：进程可寻址范围的**大小**。

合法性判定（与 OS 在软件里会做的逻辑相同，只是翻译成了硬件）：

```text
base ≤ address < base + limit
```

上界 `base + limit` 是**排他的**，故用"小于"而非"小于等于"。

实现链路：

1. 用**二进制加法器**算出上界 `base + limit`。
2. 在**地址总线**上**拦截**每一个流向内存的地址（每次访存都经过总线）。
3. 用**二进制比较器**（输出 A=B / A<B / A>B）做两次比较：`address ≥ base` 与 `address < 上界`。
4. 两个条件用 **AND 门**合并 → 末端一根导线：地址合法时才为高电平。

## 一根导线如何拦住非法访问

发地址到总线本身**不会**触发访存；真正读写要靠激活 **read enable / write enable** 信号
（读使能时内存把该地址内容放上数据总线，写使能时用数据总线的值覆写该地址）。于是最简单的拦截
办法就是**用校验电路的输出去"门控"读/写使能信号**：

- 地址越界 → 校验输出为低 → 硬件**阻断**读/写使能抵达内存模块，非法操作根本不发生。
- 再加几个逻辑门**检测**越界，向 CPU 发一个硬件信号，立即**中断**当前进程、唤起操作系统处理
  违例——通常是**终止该进程**。在 Unix 类系统上这表现为 **段错误(segmentation fault)**。

> 关键：整个校验由电路**自动**完成，OS **不需为此多执行任何指令**。OS 唯一的职责是在把 CPU
> 派给用户进程**之前，设好基址与界限寄存器**。这套电路嵌在 CPU 内部。

## 离不开运行模式与上下文切换

要让方案成立，CPU 必须支持**运行模式**（见 [[concepts/mode-bit-and-operational-modes]]）：基址、
界限这类内存管理寄存器必须是**特权寄存器**，只有内核态的 OS 能改。

这给[[concepts/context-switch]]添了一项职责：CPU 从一个进程切到另一个时，OS 不仅保存/恢复寄存器
状态，**还要把基址、界限更新为新进程的地址空间边界**。设好后 OS 把 CPU 切回**用户态**——此模式
下进程无法修改特权寄存器，于是只能用 OS 明确分配给它的那块内存。

## 与 MMU / 分页的关系

基址+界限是**最简方案**：每个进程一段**连续**物理内存。现代系统用更通用的
[[concepts/memory-management-unit]] + **分页(paging)** 取而代之——后者允许使用比物理内存更多的
内存、且不要求连续，越权访问同样经异常路径转为 [[concepts/page-fault]]/段错误。源材料把分页留作
后续主题。可以把本页看作 MMU 保护功能的"入门版原型"。
