# CPU 调度算法（Scheduling Algorithms）

调度算法本质上调度的是进程的**下一个 CPU 突发**（见 [[concepts/cpu-io-bursts]]），而非整个进程。
下面六个算法构成一条"每个修复前一个缺陷"的演进线索。

## 1. 先来先服务（FCFS）

CPU 分给第一个请求它的进程，用 **FIFO 队列**实现：进程入队尾，CPU 空闲时分配给队头。最简单，但
有**护航效应**——一个 CPU 密集型进程会让一群 IO 密集型进程排队干等，CPU 与设备利用率双低
（见 [[concepts/cpu-io-bursts]]）。

## 2. 最短作业优先（SJF）

把 CPU 分给**下一个 CPU 突发最短**的进程（同长用 FCFS 破平局），用**优先队列**实现。更精确叫
"shortest next CPU burst first"。它**可证明最优**：给出给定进程集的**最小平均等待时间**——把短进程
移到长进程前，短进程等待时间的减少多于长进程的增加。

**致命问题：无法完美实现**——无法预知下一突发的确切长度。只能**用指数平均估计**：

```text
τₙ₊₁ = α·tₙ + (1−α)·τₙ
```

`tₙ` 是最近一次实际突发，`τₙ` 是上次预测，`τₙ₊₁` 是下次预测。递归展开可知 `τₙ` 已含全部历史，
故**只需保存最近一次预测**。α 控制新旧权重：α=0 只看历史，α=1 只看最近，常用 α=½；展开后每个历史
突发乘以 `α·(1−α)ᵏ`，**越老权重按指数衰减**。无历史时初始预测设为常量（如 100ms）。

SJF 本质是一种**优先级调度**（优先级 = 预测突发的倒数），同样会导致长突发进程**饥饿**。

## 3. 轮转（Round Robin, RR）

≈ FCFS + 抢占：FIFO 就绪队列 + 一个**时间片(time quantum)**。取队头进程、设定时器、分派；然后：

- 突发**短于**时间片：进程自愿让出，取下一个。
- 突发**长于**时间片：定时器中断强制上下文切换，进程回**队尾**（回到**就绪**态而非等待态，因未请求 IO）。

> 定时器是物理器件，故抢占式调度**必须有硬件支持**（见 [[concepts/preemptive-vs-cooperative-os]]）。

RR **公平**：n 进程、时间片 q，每个拿 1/n CPU，最多等 (n−1)q 再次运行。**时间片大小是关键**：

- **太大** → 多数进程在被抢占前已进入 IO 突发，RR **退化为 FCFS**。
- **太小** → 过于频繁打断，**上下文切换激增**。上下文切换是**纯开销**，CPU 忙于切换而非执行，拖累
  **周转时间**与**吞吐量**。

## 4. 优先级调度与老化

每个进程赋一个整数优先级（范围、大小方向因系统而异，如 0–255、0–32000），用优先队列，常与 RR
结合以保证同优先级间的公平。动机：桌面上各进程并不需要同等 CPU，给交互进程更高优先级即可在不升级
硬件下提升响应性。

固有问题是**饥饿**：源源不断的高优先级进程会让低优先级进程永远拿不到 CPU。
（轶事：IBM 7094 关机时发现一个 1967 年提交、从未运行的低优先级进程。）解法是**老化(aging)**——
逐渐提高长期等待进程的优先级，使其最终也能被执行。

## 5. 多级队列（Multi-level Queue）

把所有进程放一个队列找最高优先级要 O(n) 搜索；更实用的是**每个优先级一个独立队列**。队列间也要
调度：

- **绝对优先级**：只调度最高优先级的非空队列（可能饿死低优先级队列）。
- **按优先级分配的 RR**：在队列间轮转，给不同队列分派不同数量的进程来时间片轮转，避免饿死。

常按**进程类型**分队，还可**给每队用不同算法**（如前台队列 RR、后台队列 FCFS）。这正是通用调度器
所要的：偏向交互进程但不饿死低优先级。缺点：进程**进入时被永久分配**到某队列，**不灵活**——而进程
行为会动态变化（邮件客户端后台静默 vs 切到前台要求即时响应）。

## 6. 多级反馈队列（MLFQ）

为适应动态行为而设计：

- 每个进程从**最高优先级队列**开始。
- 一个时间片内完成 CPU 突发 → 做完 IO 后**留在**该队列。
- 需要的 CPU 超过时间片 → **下移**到更低优先级队列（低优先级队列时间片**更大**）。
- 表现得像 IO 密集型的进程可**上移**。

久而久之，短突发进程留在高队列，CPU 密集型进程沉到低队列——**据行为自动调整优先级**。变体众多：每队
算法、升/降级判定标准都可不同（如用历史预测而非单次突发，避免被异常突发误导）。

**这不是在惩罚 CPU 密集型进程吗？** 两个事实化解疑虑：①IO 密集型与交互式进程都以**很短的 CPU
突发**为特征；②现代处理器极快——任务管理器里数百进程合计常用不到 10% CPU。因此**高优先级队列大多
时候是空的**，沉到底的进程不靠老化也常能很快被分派。这解释了为何渲染视频/跑基准时 CPU 飙到 100%、
桌面/音乐/浏览器却依旧流畅：调度器优先伺候日常交互任务，CPU 重活用剩余周期。MLFQ 比多级队列更
"通用"，许多操作系统历史上用过其变体。
