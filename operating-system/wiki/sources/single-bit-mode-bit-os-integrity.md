## 引言

我们学习操作系统时最先了解到的一点，就是它充当硬件与用户程序之间的**中介**。这意味着
用户程序如果没有操作系统的介入，就无法访问硬件资源。

但仔细想想，操作系统和用户程序都是软件——它们都是用编程语言写出来的。那么为什么用户
程序不能压制操作系统、夺取计算机的控制权呢？本文探讨硬件与软件之间一种巧妙的协作如何
造就了现代操作系统的健壮性。

*（注：以下内容为便于理解做了简化。文中的实现方式与汇编代码都不对应任何真实的平台或
架构。）*

这是少数几种"光靠软件无法解决、必须借助硬件支持"的场景之一。

## 特权指令与模式位

每个处理器都有一组它能执行的特定操作，称为**指令集**。如果我们想让某个软件比另一个软件
拥有更多的系统控制权，就可以定义一个并非所有进程都能使用的特殊指令子集。许多架构把它们
定义为**特权指令（privileged instructions）**。

要强制实施这一点，硬件必须支持不同的运行模式。常见做法是使用一个只存 1 或 0 的特殊单比特
寄存器，称为**模式位（mode bit）**：

- 当值为 **1** 时，处理器运行在**特权模式**，可以执行指令集中的任何指令，包括特权指令。
- 当值为 **0** 时，处理器运行在**受限模式**，无法执行特权指令。

CPU 内部用组合逻辑电路来译码指令。对普通指令（算术、数据搬移、分支、循环）而言，模式位
几乎没有影响——它们在任何模式下都能运行。但对**特权指令**而言，模式位决定了当前运行的
进程是否被允许执行它们。这样一来，特权指令只有在处理器处于特权模式时才能执行。

## 谁来翻转模式位？中断

这是整个机制中最微妙的部分：如何、由谁来翻转模式位的值？如果任何用户进程都能随意切换到
特权模式，那这个特性就毫无意义了。切换机制通常通过**中断（interrupts）**实现。

### 什么是中断

当处理器运行程序时，地址寄存器指向下一条待执行指令的内存地址。中断是发送给 CPU 的一个
信号，通知它发生了某个事件——例如用户按键或移动鼠标。当信号到来时，处理器会暂停当前
进程，跳转到某个固定内存位置，那里存放着处理该中断的代码。

这段处理例程可分为四步：

1. **保存**被中断进程的状态。
2. **处理**中断（运行处理例程，例如调用打印字符串的子程序）。
3. **恢复**被中断进程的状态。
4. 执行一条特殊指令——**中断返回（interrupt return）**，跳回被中断的进程并恢复执行。

这就是基本流程：中断临时挂起当前进程，把控制权转交给另一段例程。

### 中断翻转模式位

中断不仅能让程序计数器跳转，还能自动把模式位翻转为 1。这意味着无论那个内存位置上是什么
代码，它都会获得 CPU 的控制权——而且是在**特权模式**下。

这里有个关键点：**任何**可执行实体都可能被装载到这片内存区域——浏览器、文本编辑器，什么
都行。一旦它获得特权模式，就几乎能在计算机里为所欲为，包括确保再也没有别的进程能进入
特权模式。

现代的做法是确保**操作系统**被装载在那里，这样它就能用特权指令牢牢掌控整个系统。特权
模式本就是给操作系统内核用的，因此通常称为**内核态（kernel mode）**；受限模式是给用户
程序用的，防止它们压制操作系统，因此称为**用户态（user mode）**。

## 内核态能做、用户态不能做的事

- **I/O 设备**——操作网卡、显卡、硬盘以及显示器、键盘等外设。直接操控这些设备需要特权
  指令，所以只有内核态下的 CPU 才能做到。
- **内存 / MMU**——特权指令可以操控**内存管理单元（MMU）**，这个硬件组件负责限制 CPU
  可访问的物理内存区域。控制了 MMU 就等于完全控制了内存——这也是为什么操作系统可以读写
  物理内存的任何地方，包括用户程序的地址空间。详见 [[concepts/memory-management-unit]]。
- **重定义中断处理**——中断会让 CPU 跳到某个固定内存位置，但内存布局并非一成不变，处理
  代码不一定总在同一个地方。如果中断切到内核态后落在某个用户进程的地址空间里，那将非常
  危险。特权指令让操作系统能指定 CPU 必须跳转到的确切位置。又因为 MMU 只能通过特权指令
  操控，操作系统借此确保用户进程无法访问存放中断处理程序的内存——阻止任何恶意进程覆盖
  处理程序来夺权。

由此可见：硬件提供了工具，但最终要靠软件（操作系统）正确地使用这些工具，才能维持控制权。

用户程序在用户态下仍可使用大量指令：搬移和复制数据；算术、浮点、位运算；条件判断；以及
循环。但它们仍然需要访问硬件才有用——那它们是怎么做到的呢？

## 系统调用

为了让用户程序访问硬件资源，操作系统以库的形式提供一组函数，这些函数称为**系统调用
（system calls）**。例如要操作文件，程序需要一个系统调用来打开它，每次读写都需要更多系统
调用，最后还需要一个系统调用来关闭它。

系统调用本质上仍是函数——它们有内部实现，最终会被编译成机器码。但这看起来有点奇怪：既然
这些函数来自库，当我们使用它们时它们就成了用户程序源代码的一部分，编译后也就成了用户程序
的一部分。可用户程序运行在用户态，无法执行操控硬件所需的特权指令。

谜底是：在这些系统调用函数内部的某处，有一条触发中断的指令。**中断不仅能由硬件触发，也能
由软件触发。** 这里 `int` 指令触发一次软件 [[concepts/interrupts]]，让程序计数器跳进操作
系统的地址空间。因为中断会把 CPU 切换到内核态，该例程便能执行操控硬件、完成系统调用所需
的全部特权指令。

完成后，必须把控制权交还给被中断的进程，并把模式位重置为 0，使程序回到用户态继续运行。
具体做法因架构而异——本例由中断返回指令处理，但其他架构可能用一条专门的模式切换指令，
或显式覆写模式位后再用另一条指令返回控制权。

> 另一种理解方式：函数的一部分位于用户程序的地址空间、运行在用户态；另一部分位于操作系统
> 的地址空间、运行在内核态。

系统调用这一概念非常成功，以至于一些架构提供了专门的 `syscall` / `sysret` 指令，省去手动
设置中断的麻烦，也让汇编代码更易读。

### 优点

- **硬件抽象**——直接操控硬件既困难又复杂。系统调用替我们处理了硬件交互，让开发者用代码
  思考，而不是用晶体管思考。
- **安全性**——与抽象密切相关。直接操控硬件容易出错，可能导致漏洞和系统故障。从 Python、
  JavaScript 转到底层编程的人常问："我会不会把电脑烧坏？"答案是不会：操作系统管理硬件，
  即使写出很糟糕的 C 程序，也无法以损坏硬件的方式直接操控它。
- **可移植性**——只要多个平台遵循某些规范，同一份源代码就能在不同系统上编译。

### 缺点

- **性能开销**——中断并不便宜。它需要上下文切换等额外步骤，而缓存等现代特性会进一步抬高
  成本。此外，程序调用系统调用时其实是在中断自己、把控制权交给操作系统；一旦操作系统拿到
  控制权，并不保证立即处理该调用——它可能先去跑 CPU 调度器或管理其他资源。从某种角度看，
  **系统调用可被视为陷阱（trap）**：硬件抽象就是诱饵，引诱程序交出控制权。
- **平台依赖性**——这与前面提到的可移植性相矛盾。尽管许多平台力求兼容，仍存在一些差异
  （例如微软）。

## CPU 运行模式小结

机制很简单：用户程序可以用中断让 CPU 切换到内核态，但代价是中断自己、把 CPU 的控制权交给
操作系统。反过来，操作系统可以用一条特权指令恢复被中断的进程，但强制它继续运行在用户态。

这一概念如此成功，以至于一些架构提供专门的指令来发起系统调用，省去手动设置中断——这就是
许多架构中能见到的 `syscall` 指令和 `sysret` 特权指令。它们也让汇编代码更可读。

## 抢占式 vs 协作式操作系统

当操作系统依赖用户程序的配合来夺回控制权、执行管理任务时，它被归类为**非抢占式**（或
**协作式**）操作系统。这类通用操作系统如今已不再使用，因为有个大问题：设想一个含有死循环
且内部没有任何系统调用的程序——CPU 将永远卡在循环里。如果某进程（有意或无意）回避系统
调用，CPU 就永远不会把控制权还给操作系统，从而饿死所有其他进程。

同样，光靠软件解决不了；必须有硬件支持。解决方案是一个**定时器（timer）**，它内建于 CPU，
只能通过特权指令控制：

- 在把 CPU 分配给某个用户进程之前，操作系统设置定时器。
- 进程运行期间，定时器倒计时。
- 进程仍可通过系统调用自愿交还控制权，但若某进程运行太久，定时器到期就会触发一次硬件
  中断，强制该进程停下，把控制权交还操作系统。

这让内核态能精确掌控每个进程可使用多少 CPU 时间。再加上硬件会捕获用户态下的非法特权指令，
内核态便获得了对 **CPU 如何被使用** 的完全控制——这正是**抢占式（preemptive）**操作系统的
定义。详见 [[concepts/preemptive-vs-cooperative-os]]。

## 更宏观的图景

用户程序和操作系统都只是软件。由于用户程序被限制在用户态，无法直接与硬件交互，它们必须把
控制权交给操作系统，由操作系统代为操控硬件。所以"操作系统总是夹在中间"只是对真实情况的
一种高层抽象。

## 内核态中的驱动与第三方代码

操作系统之外的软件能在内核态运行吗？能。例如一块新显卡发布时，操作系统可能不知道如何控制
它——它是在显卡问世之前开发的。操作系统开发者无法为每一种硬件设备都写代码，于是这项任务
交给硬件厂商，通过**驱动（drivers）**完成：驱动是用来控制特定硬件的软件。由于控制硬件需要
特权指令，驱动必须运行在**内核态**。

这带来一个严重的弊端：内核态代码无法被隔离——所有内核态代码共享**同一个地址空间**
（见 [[concepts/memory-management-unit]]）。如果某个驱动误写了错误的内存地址，就可能破坏
操作系统的关键数据；如果某个驱动崩溃，会把整个操作系统一起拖垮。这就是为什么有 bug 的驱动
常在 Windows 上引发**蓝屏死机（BSOD）**。

尽管有风险，允许代码运行在内核态仍是必要的，因为驱动不可或缺。但第三方代码能在内核态运行
这一事实，催生了驱动之外的用途——比如游戏反作弊系统和恶意软件检测软件。原因有二：

- 内核态软件不依赖系统调用，因此比在用户态运行快得多。
- **反作弊示例**：设想一款游戏禁用手柄以防止对键鼠玩家不公平。某玩家可能通过第三方程序把
  手柄映射成键盘输入来作弊。由于游戏运行在用户态，无法分辨输入是否真的来自键盘；而内核态
  的反作弊软件可以直接监控硬件活动，发现输入其实来自手柄。
- **恶意软件检测**：内核态让软件能扫描网卡之类的设备、监控网络流量，并检查运行中进程的
  地址空间，以排查可疑行为——例如窃取私密数据并发送到不可信的端点。

### 当内核态软件出问题时

什么都用内核态并不总是好主意。以 **[[entities/crowdstrike]]** 为例——这是一款安装在全球
数百万台企业设备上的恶意软件检测软件。在九月[^crowdstrike-date]，一次更新损坏了一个配置
文件，将其填满了零；文件损坏后软件崩溃，进而导致操作系统也一起崩溃。结果是全球范围的混乱：
数千台设备显示蓝屏死机，企业损失数十亿美元。

另一个例子是 **[[entities/vanguard]]**——Riot Games 的反作弊软件，用于《英雄联盟》《无畏
契约》等热门游戏。刚发布时，它曾导致数百名玩家蓝屏死机。但更可怕的是它与 **[[entities/tencent]]**
（拥有 Riot Games 的中国公司）的关联。事实证明，不安装 Vanguard 就无法玩 Riot 的游戏，这
对反作弊而言合情合理；恼人的是，Vanguard 即使在你不玩游戏时也运行在内核态，意味着它可以
监控你在计算机上做的一切。尽管没有证据表明它确实这么做，但腾讯作为一家很可能受政府监管的
中国公司，加之 Vanguard 运行在全球数百万台计算机上，这一点令人颇为不安。
