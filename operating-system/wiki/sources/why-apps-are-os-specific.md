## 引言：不只是架构问题

把一个可执行文件从 Windows 拷到 Mac，它跑不起来。常见解释是"Mac 用 ARM、Windows 用 Intel
x86"——合理但不完整。因为：

- 回到 2019 年，几乎所有 Mac 还在用 Intel，**同样跑不了** Windows 程序。
- 你甚至能在**同一台机器**上装 Linux 和 Windows(同一 CPU)，但 Linux 上**仍然跑不了** Windows
  程序。

既然编译后的代码就是 CPU 能执行的机器指令，同一 CPU 上为何不能两边通用？答案是：**应用程序不仅
与架构绑定，也与操作系统绑定**。(作者注：如果这个限制不存在，世界会美好得多。)

## 第一层原因：系统调用

向计算机科学毕业生提这个问题，多半会回答**系统调用(system calls)**。快速回顾(详见
[[concepts/system-calls]] 与 [[concepts/mode-bit-and-operational-modes]])：

- 通用操作系统要同时运行多个进程、共享硬件(尤其是 CPU)。任何进程理论上都可能利用 CPU 夺取系统
  控制权，绕过操作系统作恶。
- 为防止这点，CPU 有两种基本模式：**内核态(privileged mode)** 可执行指令集中任何指令(操控 IO
  设备、CPU 定时器、MMU 等)；**用户态**只能执行受限子集(搬运/复制数据、算术、条件判断、循环)，
  **不能执行特权指令**。
- 用户程序跑在用户态，操作系统跑在内核态。于是即便"读个文件""发条网络消息"这种事，用户程序也
  得**请操作系统代劳**——这就是系统调用：**操作系统所提供服务的接口**。

系统调用通常以 C/C++ 函数形式提供，大致分为**六大类**。**每个操作系统提供自己独有的一套系统
调用**——兼容性问题正是从这里开始。

实际中几乎所有程序都重度依赖系统调用，连你现在看视频的播放器每分钟都在发出**成千上万次**请求。
在 Linux 上可用 `cat /proc/<pid>/status` 查看进程的 `voluntary_ctxt_switches` 字段，它约等于
进程主动调用操作系统的次数——数字不小。

**核心问题**：用户程序对系统调用依赖极深，只要另一个操作系统没有**以完全相同的方式**实现它们，
程序就跑不了。

### 不只是命名差异：fork vs CreateProcess

即便两个系统的系统调用有"对应物"，**功能也可能差很多**。例子：

- **Windows** 的 `CreateProcess`：接收可执行文件路径，直接**新建一个进程**运行它。
- **Unix** 的 `fork`：也创建新进程，但创建的是**调用进程的克隆**(只有 PID 不同)。要运行一个
  特定程序，`fork` 之后子进程还得再调用 **`exec`**，用目标可执行文件的内容**替换自己的 text 段**。

(这个话题很有意思，或许值得单独一集。)于是同一件事：一个平台一次系统调用、另一个平台要好几次。
**编译成机器码后，即使针对同一 CPU 架构，两边也会生成不同指令**；这些多出来的指令在另一个操作
系统上毫无意义，导致未定义行为乃至崩溃。

## 第二层原因：ABI(应用二进制接口)

就算两个系统定义了**完全相同**的系统调用集合，更底层的实现细节仍会造成不兼容。

程序请求操作系统时会触发一次**中断**，让 CPU 切到内核态、把控制权交给操作系统。在 x86-64 上用
一条专门指令 **`syscall`** 实现。但操作系统接管后，怎么知道请求的是**哪个**系统调用?参数又如何
从用户程序传给操作系统?

- **识别**：给每个系统调用配一个**唯一编号**。调用前程序把编号写进某个**寄存器**；操作系统接管
  后读该寄存器，并查**系统调用表**确定是哪个调用。(在 Linux 内核源码的架构相关文件里能查到每个
  系统调用的编号，也能看出 Linux 支持的系统调用数量之多——前面讨论的只是最常用的一小部分。)
- 即使两系统用**相同的系统调用表**，若一方用**不同的寄存器**放调用号，程序就会读错寄存器、认错
  系统调用 → 未定义行为。
- **传参**：即便读同一个寄存器拿调用号，许多系统调用还需参数，而**传参方式因操作系统而异**——
  有的放寄存器、有的写进特定内存区域、有的两者混用。例如 **Linux**：参数少于 6 个时放寄存器，
  超过 6 个则其余写入内存。
- 参数放内存时，**方式**也有讲究：用栈还是某个专门内存位置?若参数含**指针**，**地址宽度**(32/64
  位)也变得关键。

这一整套需要对齐的底层约定就叫 **ABI(Application Binary Interface,应用二进制接口)**。正如 API
在应用层定义函数,**ABI 定义二进制代码各组件在特定操作系统与架构上如何交互**。详见
[[concepts/application-binary-interface]] 与 [[concepts/syscall-abi]]。

## 第三层原因：可执行文件格式

而且这还只是系统调用层面的事。即使上面的问题都解决了,还有别的底层细节会造成不兼容,其中最重要
的之一是**可执行文件格式**。

编译器翻译源代码后产出可执行文件,它**不只含指令**,还含**数据**与**元数据**,供操作系统加载并
执行程序:程序的各部分(在这一堆 0/1 里)分别位于何处?长什么样?**每个操作系统对此都有自己的
规则**,这套结构保证可执行文件能被正确加载运行,并支持链接、调试、动态库等特性。代表:

- **ELF**(Executable and Linkable Format)——Linux 使用。
- **PE**(Portable Executable)——Windows 使用。

详见 [[concepts/executable-file-format]]。

## 第四层原因：运行时环境

另一个原因是**运行时环境(runtime)**。有些语言不直接编译成机器码,而是跑在**虚拟机或解释器**上;
只要目标系统装了对应的虚拟机,同一份代码就能跨平台运行。讽刺的是,**虚拟机/解释器恰恰是为了绕开
本视频讨论的诸多兼容性问题而设计的**。

但现代应用常是**模块化**的:若所有模块都编译成机器码当然好;可只要有**一个模块依赖某运行时**而
该运行时未安装,整个应用就可能无法运行。(作者吐槽:不反对需要运行时的语言,但也许不该用它们写
系统的关键组件——话说回来,我们可是在服务器上跑 JavaScript,所以也没立场抱怨。)

## 小结

应用程序与操作系统绑定,是多层底层约定叠加的结果:

1. **系统调用集合不同**——名字与功能都可能不同(`fork`+`exec` vs `CreateProcess`)。
2. **ABI 不同**——即便系统调用表相同,调用号所在寄存器、传参方式、指针地址宽度等约定也须对齐。
3. **可执行文件格式不同**——ELF vs PE,操作系统据此加载程序。
4. **运行时依赖**——依赖未安装的虚拟机/解释器会导致失败(反过来也是跨平台的一种手段)。
