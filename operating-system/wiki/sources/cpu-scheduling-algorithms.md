## 引言：为何满载也不卡

跑基准测试时 CPU 占用飙到 100%，系统却依然可用——这是因为通用操作系统**优先保证响应时间
(response time)**，让交互式应用即使在重负载下也保持流畅。本集讲的就是实现这一点的核心组件：
**CPU 调度器(scheduler)**。

CPU 调度是操作系统根据特定**标准**决定某一时刻把 CPU 分给哪个任务/进程的技术。常见标准有：
响应时间、CPU 利用率、吞吐量、周转时间、等待时间。

## 把 CPU 分配给进程意味着什么

程序运行后成为**进程**，操作系统为其保留一块内存，其中的**文本段(text section)** 存放可执行
代码，每个进程有自己独立的文本段。CPU 里的**地址寄存器（程序计数器 PC）** 指向下一条要执行的
指令。所谓"把 CPU 分配给某进程"，就是让 PC 指向该进程的文本段；切换到另一进程则要修改 PC，
这一更复杂的操作就是[[concepts/context-switch]]。

进程**不是**传统意义上的对象或结构体实例，而是"计算机运行所处的上下文"。操作系统用一个叫
**进程控制块(PCB, Process Control Block)** 的结构来**表示**进程，记录其状态、PC、寄存器等
管理数据。每创建一个进程就生成一个 PCB。本文用方框代表 PCB，常口头称之为"进程"，但进入队列
的其实是 PCB。

## 最简单的算法：先来先服务(FCFS) 及其天真假设

FCFS 把 CPU 分配给第一个请求它的进程，用一个 FIFO 队列即可实现：进程创建时 PCB 入队尾，CPU
空闲时分配给队头。但若假设"进程必须执行完才轮到下一个"，会有两大问题：

1. **并非所有进程都会终止**——服务器、后台服务设计为永久运行；若把 CPU 给了不结束的进程，后面
   的进程将永远等待。
2. **运行中的程序并不连续使用 CPU**——一个耗时 2 分钟的任务，绝不是把 CPU 占满 2 分钟，中间常在
   等待别的东西。

## CPU 突发与 IO 突发

以"分块复制文件"的程序为例：读写文件要调用操作系统、与硬件(磁盘)交互，即 **IO 操作**。IO 很慢，
其耗时取决于 IO 硬件而非 CPU。于是程序对 CPU 的使用呈现空隙：

- **CPU 突发(CPU burst)**：程序占用 CPU 计算的时段。
- **IO 突发(IO burst)**：程序等待 IO 完成的时段。

每个进程的执行都是 CPU 突发 → IO 突发 → CPU 突发 → … 交替，最后以一次"终止执行"的系统调用
结束。在冯·诺依曼架构(占现代计算机 99% 以上)中这些空隙是常态而非例外。

**CPU 突发时长的分布**经数十年测量呈现规律：**绝大多数是短突发，长突发很少但概率不为零**。
这是设计调度算法的关键观察——进程大量时间在等 IO，所以当一个进程进入 IO 突发时，理想做法是
把 CPU 让给另一个不在等 IO 的进程，**尽量让 CPU 保持忙碌**。因此调度器应只把 CPU 分配到
**当前 CPU 突发结束**为止，而非到进程完全终止。更精确地说，FCFS 其实是"first CPU burst come
first served"。

## 进程状态

考虑到上述行为，进程会处于不同状态：

- **新建(new)**：程序刚启动，可执行文件还在载入内存。
- **就绪(ready)**：已载入并分配好资源，等待 CPU 时间。
- **运行(running)**：CPU 已分配给它。
- **等待(waiting)**：进程发出 IO 等系统调用，自愿交还 CPU，等待事件发生或 IO 完成（请求资源
  **或**单纯等待按键、网络请求等事件）。完成后回到就绪态——注意不会立即恢复运行，因为期间 CPU
  多半已给了别人，它必须等轮到自己。
- **终止(terminated)**：多数用户程序最终到此；某些系统还细分"正在终止"与"已完全释放资源"。

进程一生大多在 **就绪↔运行↔等待** 三态间循环。这正是队列成为调度核心数据结构的原因：除运行态
外的进程都在等待，而同一时刻一个核心上只能有一个进程运行（运行进程不入队）。

## 调度器 vs 分派器(dispatcher)

- **调度器(scheduler)**：按调度策略管理就绪队列，**决定**下一个用 CPU 的进程。
- **分派器(dispatcher)**：**执行**这个决定——做上下文切换：把当前 CPU 状态存入被中断进程的
  PCB、放入相应队列，再取就绪队头的 PCB 恢复其寄存器与 PC、重分配 CPU（若硬件支持还要切回用户
  态）。

分派器本身也要消耗 CPU 来做上下文切换，这段延迟叫**分派延迟(dispatch latency)**。因为每次上下
文切换都调用它，所以它是操作系统中被高度优化的组件之一。本文用的图是**甘特图(Gantt chart)** 的
变体，展示各进程的起止时间。

## FCFS 的护航效应(convoy effect)

CPU 突发频率图显示长突发虽罕见但概率非零。据此把进程分为两类：

- **IO 密集型(IO-bound)**：大量短 CPU 突发，性能受 IO 子系统速度制约（最常见）。
- **CPU 密集型(CPU-bound)**：少量长 CPU 突发，受 CPU 速度制约。

当一个 CPU 密集型进程长时间霸占 CPU 时，其它 IO 密集型进程做完 IO 后全堆在就绪队列里干等，
此时 IO 设备闲置；等大进程终于让出、小进程飞快跑完又回去做 IO，CPU 又空了下来——大进程再回来
独占。所有进程为一个大进程让路，这就是**护航效应**，导致 CPU 与设备利用率都偏低。让短进程先走
能缓解它。

## 最短作业优先(SJF)

SJF 把 CPU 分给**下一个 CPU 突发最短**的进程（同长时用 FCFS 破平局），可用**优先队列**实现。
更精确叫"shortest next CPU burst first"。它**可证明最优**——对给定进程集给出**最小平均等待时间**：
把短进程移到长进程前，短进程等待时间的减少多于长进程等待时间的增加，平均等待时间因而下降。
最小化平均等待时间很关键，因为那是进程"就绪但拿不到 CPU"白白浪费的时间。

**致命问题：无法完美实现**——除非能预知未来，否则无法知道下一个 CPU 突发的确切长度（循环、函数
调用、用户输入都会改变它）。只能**估计**：假设下一次突发与历史相近，用**指数平均**预测：

```text
τₙ₊₁ = α·tₙ + (1−α)·τₙ
```

- `tₙ` 是最近一次**实际**突发，`τₙ` 是上次的**预测**，`τₙ₊₁` 是对下一次的预测。
- 递归展开可知 `τₙ` 已包含全部历史，所以**只需保存最近一次预测**，无需存全部历史。
- **α 控制新旧权重**：α=0 完全忽略最近突发(只看历史)；α=1 只看最近突发(忽略历史)；常用 α=½
  对新旧等权。展开后每个历史突发被乘以 `α·(1−α)ᵏ`，因 0<α<1，**越老的突发权重越按指数衰减**。
- 进程刚创建无历史时，初始预测设为常量(如 100ms)或系统平均值。例如以 100ms 起步，预测的下一突发
  约为 72ms——比最老的突发更接近近期突发。长期来看，这种预测虽不完美，但与实测突发吻合得不错。

## 抢占 vs 非抢占；饥饿

进程 A 正在运行，进程 B 进入就绪队列且其下一突发比 A 剩余突发更短——要不要打断 A？

- **抢占式(preemptive)**：操作系统可以打断正在运行的进程。
- **非抢占式(non-preemptive)**：绝不打断，B 必须等 A 让出 CPU。

抢占是现代系统的默认选择，因为**非抢占无法对付死循环**：若某进程拿到 CPU 后陷入不含任何系统
调用的无限循环，就永远不交还 CPU，所有其它进程从用户视角看全部冻结——这违背了并发（让 CPU
极快地在进程间轮换、制造"同时运行"错觉）这一通用系统的核心特性，恶意软件还能借此故意垄断 CPU。

任何调度策略导致某进程在就绪队列中**无限等待**，就叫**饥饿(starvation)**。SJF 里一个长突发的
进程可能被源源不断到来的短突发饿死。饥饿不只与突发长短有关，而是**调度策略本身**的后果。SJF 虽
在利用率和等待时间上高效，但通用系统还需要别的标准。

## 轮转(Round Robin)

RR ≈ FCFS + 抢占：FIFO 就绪队列 + 一个**时间片(time quantum/time slice)**（动画里用 100ms）。
新进程入队尾；调度器取队头进程、设一个定时器、分派它。然后两种情况：

- 突发**短于**一个时间片：进程自愿让出，调度器取下一个。
- 突发**长于**一个时间片：定时器到期触发中断，强制上下文切换，进程被放回**队尾**（注意它是回到
  **就绪态**而非等待态，因为它没请求 IO）。

> 定时器是物理器件（通常嵌在 CPU 电路里），所以**抢占式调度必须有硬件支持**。

RR 能**公平**分配：n 个进程、时间片 q，每个进程拿到 1/n 的 CPU（每块至多 q），且一旦入队最多
等 (n−1)q 就能再次运行。但等待时间未必最优，**时间片大小是关键**：

- 时间片**太大** → 多数进程在被抢占前就进入 IO 突发，RR **退化为 FCFS**。
- 时间片**太小** → 进程被过于频繁打断，**上下文切换次数激增**。而上下文切换时间是**纯开销**
  （切换时系统不做有用功），CPU 忙于切换而非执行。

由此引出更多标准：

- **周转时间(turnaround time)**：从进程创建到完成的总时间 = 就绪队列等待 + CPU 执行 + IO 时间。
- **吞吐量(throughput)**：单位时间完成的进程数。
- **响应时间(response time)**：从提交请求到产生**第一个响应**的时间（不是完成时间）。它直接影响
  交互系统的"流畅感"，尤其在多任务时。

平均周转时间不一定随时间片增大而改善；一般当多数进程能在一个时间片内完成其 CPU 突发时较好，
以保证上下文切换只在必要时发生。良好的算法要在 CPU 分配与开销间平衡，兼顾周转时间和吞吐量。

## 优先级调度与老化(aging)

RR 是实现并发最简单的方式之一，但处理器有速度上限：并发进程太多时响应时间变长、"同时运行"的
错觉破灭；缩小时间片又会带来过多上下文切换。三种解法：①提高处理器速度、②增加处理器（前两者要
硬件升级，未必可行）、③**用更聪明的调度器**（纯软件即可）。

观察：你同时跑游戏、直播、邮件、文件管理器……这些进程**并不需要同等的 CPU 时间**。给合适的进程
更高优先级，就能在不升级硬件的前提下显著提升整体响应性（邮件通知晚半秒无所谓，但操作系统服务等
后台进程是关键的）。

**优先级调度**用优先队列，每个进程赋一个整数优先级（范围、以及"小值=高优先级还是低"都因系统而异，
如 0–255、0–32000）。常与 RR 结合，保证同优先级进程间的公平。

固有问题仍是**饥饿**：重负载下源源不断的高优先级进程会让低优先级进程永远拿不到 CPU。SJF 本质就是
一种优先级调度（优先级 = 预测下一突发的倒数）。

> 轶事：IBM 7094（1962 年，NASA/美空军用）于 1973 年关机时，发现一个 1967 年提交、从未运行过的
> 低优先级进程。

解法是**老化(aging)**：逐渐提高长期等待进程的优先级。例如优先级 127(低)~0(高)，每秒把等待进程
优先级 +1，最终连初始 127 的进程也会升到最高被执行。

## 多级队列与多级反馈队列

优先级 + RR 若全放一个队列，找最高优先级进程可能要 O(n) 搜索。更实用的是**为每个优先级设独立
队列**，即**多级队列(multi-level queue)**。队列间也要调度：

- 一种是**绝对优先级**：高优先级队列对低优先级队列有绝对优先，只调度最高优先级的非空队列。
- 为避免饿死低优先级队列，也可**按优先级以 RR 方式在队列间分配不同数量的进程**来时间片轮转。

多级队列常按**进程类型**分队，还可**给每个队列用不同算法**（如前台队列用 RR、后台队列用 FCFS）。
这正是通用调度器所期望的：给交互式进程优先权，但不至于饿死低优先级进程。但缺点是进程**进入系统时
被永久分配到某队列**，不灵活——而进程行为会**动态变化**（邮件客户端平时在后台静默，被切到前台时
却要求即时响应）。

**多级反馈队列(MLFQ, multi-level feedback queue)** 用于适应这种动态变化：

- 每个进程从**最高优先级队列**开始。
- 若在一个时间片内完成 CPU 突发，做完 IO 后可留在该队列。
- 若需要的 CPU 时间超过时间片，则**下移到更低优先级队列**（低优先级队列的时间片更大）。
- 表现得像 IO 密集型（在更高队列的时间片内就完成突发）的进程可被**上移**。

久而久之，短突发进程留在高优先级队列，CPU 密集型进程逐渐沉到低优先级队列——系统据行为**自动**调整
优先级。MLFQ 有很多变体：每队的算法、升/降级的判定标准都可不同（如用历史预测而非单次突发来避免被
异常突发误导）。

**这不是在惩罚 CPU 密集型进程吗？** 两个事实化解了疑虑：①IO 密集型与交互式进程不完全相同，但都以
**很短的 CPU 突发**为特征；②现代处理器极快——任务管理器里几百个进程合计常用不到 10% CPU。因此**高
优先级队列大多时候是空的**，沉到最低队列的进程不靠老化也常能被很快分派。这就解释了为何渲染视频/跑
基准时 CPU 飙到 100%，桌面、音乐、浏览器却依旧流畅：调度器优先伺候日常交互任务，CPU 重活用剩余周期。

## 收尾澄清

- **现代系统调度的是线程而非进程**，使单个应用内也能并发（一个线程管 UI、另一个做后台计算）。配合
  MLFQ，CPU 重活进程的 UI 也能流畅——管 UI 的线程可独立保持高优先级。见 [[concepts/threads]]。
- 本文把 **IO 队列**简化成 FIFO，实际 IO 子系统更复杂，进程完成顺序可能与到达顺序不同；FCFS 对 IO
  设备同样低效（要用显卡的进程不该排在用磁盘的进程后面）。这属于另一个组件——**IO 调度器**。
- 后续会专门讲 Linux、Windows、BSD 等现代系统的调度策略。
