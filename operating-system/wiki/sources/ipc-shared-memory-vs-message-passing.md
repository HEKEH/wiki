## 引言

视频开场展示了一个看起来像图片、实则是程序的画面：用 **Rust** 写的部分负责计算分形中每个
像素的颜色，把数据写进一个数组；全部算完后触发一个事件，把数组作为**消息**发出。
**TypeScript** 部分监听该事件、接收数据，再交给函数把像素画到 HTML canvas 上。

像这样**模块化**构建的应用之所以可能，正是得益于**进程间通信（IPC）**。本文讨论默认相互
隔离的进程，如何通过交换信息与其他进程交互。

## 进程为何需要协作

在前一集里我们知道：程序被加载进内存开始执行后会获得一个新名字——**进程**。但进程不只是
载入内存的可执行代码，它还包括 CPU 状态、分配给它的内存区域、打开的文件列表，以及 IO 设备
等资源。因此进程是程序运行所处的整个**上下文**。

这种隔离带来一个明显的缺点：很多场景下进程需要与其他进程协作。如果进程完全隔离，就无从协作。
据此可把进程分为**独立进程（independent）**与**协作进程（cooperating）**。

促成协作的理由：

- **共享数据**——两个共享数据的进程显然是协作进程。
- **计算加速**——在支持并行的系统上，把复杂任务拆成可同时执行的小任务，缩短总耗时。
- **模块化**——把系统功能拆分到不同进程中。

无论哪种情况，协作进程都需要一种**进程间通信**机制来交换数据。IPC 有两种基本模型：
**共享内存（shared memory）**和**消息传递（message passing）**。

## 共享内存

操作系统为每个进程分配独立的内存块，即进程的**地址空间**；并通过**特权指令**机制强制隔离：
进程若试图读写另一个进程的地址空间，操作系统会立即中断并终止它，以保证数据安全。

因此要让多个进程共享一块内存，必须让操作系统解除这一限制：

1. 通常用**系统调用**创建一块共享内存区域，它位于创建者进程的地址空间中。
2. 任何想通过该区域通信的进程，也要用系统调用把它**附加（attach）**到自己的地址空间。
3. 建立完成后，双方就能通过读写这块共享区域来通信。

**关键细节**：一旦操作系统授予了共享内存区域，它就不再管理进程拿它做什么。数据如何组织、
写到区域内的哪个位置，完全由进程自己决定，与操作系统无关。

以**生产者—消费者**模型为例：若共享区域里是一个数组，生产者写入 8 位有符号数，而消费者却按
8 位无符号数读取，则同样的比特会被赋予不同含义；若消费者期望的是别的数据类型（如 32 位无符号
整数）或从错误的地址读取，就会发生误解释。轻则一方或双方失败，重则**未定义行为**。

此外，进程还要**自己负责**避免同时写入同一位置，否则会遇到**竞态条件（race condition）**——
这属于线程同步的话题。

**使用共享内存的程序**：任何基于 Chromium 的浏览器都是模块化系统的典型例子。为防止网页
JavaScript 的 bug 拖垮整个浏览器，Chromium 用共享内存把任务分给三类进程：

- **browser 进程**——管理用户界面和 IO 操作，Chrome 启动时只创建一次。
- **renderer 进程**——处理网页内容（HTML、JavaScript、图片），每开一个标签页就新建一个。
- **plug-in 进程**——管理 Flash、QuickTime、PDF 阅读器等具体插件。

这种隔离保证了某个标签页崩溃时只有它的 renderer 进程失败，其他标签页不受影响。其他例子还有
仿真软件、游戏引擎、数据库管理系统和深度学习框架。

## 消息传递

共享内存并非总是最佳方案——它容易出错，且把过多责任压在程序员身上。另一种做法是由操作系统
提供机制，让进程在**不共享地址空间**的前提下通信与同步：地址空间保持隔离，进程通过**发送
消息**来通信。常见机制有**管道（pipes）**、**套接字（sockets）**和**远程过程调用（RPC）**。

一般概念如下。设有进程 A 和 B，A 想给 B 发消息：

1. A 发起一个系统调用，请求操作系统与 B **建立链接（link）**——这是消息传递的逻辑路径。
2. 实际上内核在自己的地址空间里创建一个**队列（queue）**，充当 A 发、B 收的**邮箱（mailbox）**。
3. 队列行为可按需求变化：同步或**异步**通信、**带缓冲**队列（限制邮箱可容纳的消息数）。
   要**全双工**通信，邮箱可用两个队列实现，让双方能同时互发而不冲突。

**这里有个微妙之处**：邮箱位于操作系统的地址空间，进程本不能直接访问它。因此操作系统必须为至少
两个操作提供系统调用——**send** 和 **receive**：

- A 要发消息时，用系统调用告诉操作系统"我无法直接访问与 B 共享的邮箱，请帮我把这条消息放进去"。
  因邮箱在内核地址空间，操作系统可把消息**复制**进邮箱。
- B 要查消息时，用对应系统调用请操作系统检查邮箱；若有消息，操作系统把它作为函数返回值返回。

**Mach 与端口**：最早引入共享邮箱概念的操作系统之一是 **Mach OS**，其衍生系统是 iOS 和
macOS 等现代系统的基础。Mach 把进程当作**任务（task）**，并把邮箱称为**端口（port）**。要点是：
消息发往**端口**而非直接发往进程。因为两个进程可以有多条通信链接，而且并非所有端口都恰好关联
两个进程——进程可保留一个开放端口接收任意其他进程的消息，这种端口称为**监听端口（listening
port）**，用于处理建立新私有链接的特殊消息**连接请求（connection request）**。

**跨机器通信**：消息传递的巨大优势是不要求两个进程共享地址空间，因此进程甚至**不必在同一台
机器上**。这需要更复杂的底层实现（网络能力、网卡的驱动级编程），但若在操作系统层面正确实现，
对开发者应是无缝的——跨机器的消息传递与同机器的用法相同。

**客户端—服务器与套接字**：客户端—服务器架构通常用**套接字接口**实现，这是几乎所有主流操作
系统都支持的消息传递机制。注意客户端与服务器其实是**进程**而非机器，通信也不一定需要网络
（如用 localhost 测试时两者在同一台机器）。客户端向服务器发请求时，**IP 地址**标识承载服务器
进程的机器，**端口**代表服务器进程接收请求的邮箱。提供 HTTP、FTP、SSH 等服务的服务器都依赖
套接字接口。

## 两种模型的权衡

消息传递的**缺点**：端口位于内核地址空间，进程每次收发消息都要走系统调用，性能开销较大。

共享内存则没有这一限制：系统调用只在创建共享区域和附加进程时才需要；之后进程读写共享区域就
像访问自己地址空间一样，无需再走系统调用。因此共享内存的通信极快——基本等同于直接内存访问。

> 澄清：并不是说消息传递慢，只是共享内存更快。99% 的场景下消息传递已经足够。
