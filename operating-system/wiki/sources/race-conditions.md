## 引言：当并发实体需要交互

并发是这个频道反复出现的主题——它与 CPU 调度一起，把计算资源分配给多个可执行实体。系统在它们之间
极快地切换 CPU，制造出"全部同时运行"的错觉。当这些实体**各自独立**时一切顺畅；可一旦并发实体需要
**交互**，事情就变得棘手，处理不当后果可能是灾难性的。本集讲**竞态条件**——系统编程中最离奇、最难
捉摸的 bug 之一。

## 第一关键概念：大多数操作不是原子的

初学时我们以为写代码就是一条接一条的指令，但**每行代码未必只对应一条 CPU 指令**。例如把两个数相加，
需要：把每个操作数从内存取入寄存器、令 CPU 相加、再把结果存回内存。作为系统程序员必须牢记：我们写
的多数操作都需要**多个 CPU 步骤**才能完成——**大多数操作不是原子的**。

## 复习：线程与并发

线程通常通过系统调用创建。在 C 里最常见的是 POSIX 系统上的 `pthread_create`，它接收一个函数指针。
调用后底层系统调用告知操作系统：该函数可作为独立可执行实体、与调用线程并发运行；于是 `main` 不等
新线程结束就继续往下跑。

注意**并发不等于并行**：多核上并发函数可能真正并行，单核上则共享一个核心、各线程轮流，切换快到看似
同时（见 [[concepts/parallelism]]）。

线程与并发用途极广，例如现代 HTTP 服务器靠线程同时处理多个客户端请求：主线程监听请求，一到达就派生
新线程去处理，自己立刻回去继续监听。若线程间**不共享数据**（如各自只回显自己客户端的名字），基本
不会出错。

## 出问题的例子：共享字符串

把服务器改成回复一个**所有线程共享**的字符串，该串由另一个 writer 线程周期性地从一组句子里循环更新。
作者的天真实现：定义全局变量存当前串，一个 writer 线程循环改它，handler 线程在响应时读它，并把每次
响应都记进日志。

乍看一切正常。但在 **9 百万**请求里发现 **1 条**响应是一个列表里根本不存在的串——最后一个字母不对。
单次不足以下结论。于是跑更久，在 **8500 万**请求里找到更多可疑输出：给每句染不同颜色后可见，有些响应
是**不同句子的片段拼接**而成。

## 原因：字符串操作不是原子的

C 字符串在内存中是以 null 结尾的字符序列（这省去了同时存指针和长度）。关键在于多数字符串操作非原子：
打印要逐字符输出到 null；拷贝要**逐字符循环**复制到 null（即便按块拷贝，块也有最大尺寸，长串仍需多
步）。

于是：若一个线程正读共享串、另一个正写它，并发可能让操作系统在 writer **拷到一半**时把 CPU 切给别的
进程。此刻共享串处于**无效状态**——值是我们从未打算让它出现的样子。若调度器又在 writer 恢复并写完前，
把 CPU 给了一个去读该串的线程，它就会拷走**不一致的数据**。这正是日志里那些"由不同有效句子片段拼成的
乱串"的由来。

## 竞态条件的定义

**竞态条件**是一种软件 bug：程序结果取决于多个线程/进程访问与修改**共享资源**的**不可预测顺序**。
它通常是**抢占式操作系统**的副作用——"不可预测"正源于此。抢占式系统会不时强行打断运行中的进程，让
别的进程用 CPU，以公平分配计算资源。

回顾 CPU 调度里的**时间片(time quantum)**：进程/线程能连续用 CPU 的固定时长。线程可经系统调用主动
让出；若任务未在时间片内完成，硬件定时器触发中断**抢占**它，OS 暂停它并把 CPU 分给别处。

所以**若一次数据修改耗时超过一个时间片，数据必然进入无效状态**——至少保证被打断一次。但更短的操作
也不安全：①操作起于时间片末尾时极易被打断；②CPU 还会被许多与调度器无关的事件中断——其他进程设的
定时器、用户按 UI 按钮、到来的网络包……**任何中断不论来源都能抢占当前线程**，哪怕时间片没到期。我们
无法阻止这些事件（否则系统就对它们失去响应）。但能否至少防止它们破坏数据？

## 尝试一：用控制变量当标志（失败）

引入一个标志：1=正在写，0=未写；读者读串前先查标志。**但标志一旦被线程共享，它本身也成了共享数据、
同样会竞态**。设置/检查变量并非总是原子：某些架构上写变量可能原子，但**读取并比较几乎从不原子**——
要把值从内存载入寄存器、与期望值比较、再据结果行动。

设想：读者线程查标志，把值载入寄存器、刚比较完就被中断；writer 线程随后把标志置 1 并开始改串、也被
中断；读者再被分派、状态恢复，于是带着寄存器里**过期的旧标志值**继续，误以为"安全"而读了处于无效
状态的串——变量没帮上多少忙。

所以除非很清楚自己在做什么，否则应避免用普通变量当标志来控制对共享数据的访问。像 **Peterson 算法**
这类经典软件方案依赖许多微妙假设（软件行为、编译器优化、乃至硬件架构）才能在特定情形下正确工作——
细节留到线程同步那一集。

## 多写场景：丢失更新

除了"一写多读"，还有**多个线程同时写**共享数据的情形，竞态同样危险。例如并行统计 0 到 N 之间的质数
个数，多线程各跑一个核，用一个**共享计数器**，每找到一个质数就自增。

问题：很多架构上**自增不是原子操作**。

- **并行下**：两线程几乎同时找到质数，本应让计数到某值，却可能都读到同一旧值、各自自增、再写回，
  结果只 +1——**无需任何中断**，纯由并行访问共享变量所致。
- **无并行也会坏**：已有 10 个质数，五个线程几乎同时各找到一个，期望变 15。设第一个线程取值、自增，
  在写回前被中断；其余四个无中断地依次自增，计数到 14；第一个线程恢复，从"写回"那步继续，但它持有的
  是被其他线程修改**之前**算出的值，于是**覆写**掉更新，计数从 14 跌回 11——四个正确探测到的质数就这样
  丢了。

计算机其实"很笨"：它不识别这些情形，只盲目执行指令，哪怕指令在制造问题。更诡异的是，同一程序多次运行
**几乎每次结果不同**，有时甚至碰巧正确——因为行为取决于不可控事件的时机与顺序，无从预测。

## 收尾：最危险的是"罕见"

竞态条件最危险之处恰恰是它的**稀有**：作者的服务器例子 8500 万请求里只 14 次。概率极低，因而**极难
复现和调试**。下次项目用到并发就记住：若不小心，你派生的每个线程都可能在给一颗定时炸弹倒计时——最好
的情况是崩溃，最坏的情况无从预料。所幸计算机提供了防止竞态的工具（同步），那是未来一集的主题。
