## 问题：被中断进程的状态不能丢

进程运行时不断用 CPU 逐条执行指令，而**一条指令的结果取决于执行时的整个上下文**——尤其是
**寄存器的状态**。同一条指令，寄存器里的值不同，结果就不同。

所以在并发场景下，一个进程被中断时，**不能直接**把 CPU 交给下一个进程：此刻寄存器里的数据
属于**被中断的进程**。若新进程不加处理就接着跑，就会在**错误的上下文、用错误的数据**运行。

因此每次为把 CPU 分给别的进程而中断一个进程时，都必须：把当前寄存器状态**捕获并存入内存**，
再**载入新进程的状态**让它正确继续。这就是**上下文切换(context switch)**——因为我们改变了
CPU 运行所处的"上下文"。

关键在于：这**不会自动发生**，操作系统得**真的跑一段代码**来完成它。本视频回答一个常被追问的
问题：**如何在不破坏被中断进程状态的前提下完成这件事？**

## 前提：基于中断的切换

本文讨论的是**基于中断**的切换。在不提供中断机制的极简 CPU 上，上下文切换**完全靠软件**——
每个程序在让出 CPU 前负责**保存自己的状态**(即协作式，见
[[concepts/preemptive-vs-cooperative-os]])。这类 CPU 很罕见，因为既不安全也不实用。

**中断(有时也叫陷阱 trap)是计算机的基本特性**：近 50 年历史的传奇 MOS 6502 就有中断；连用于
控制电子设备的简单微控制器也有；带中断机制的计算机已存在 70 多年。

进程被中断有两种方式(见 [[concepts/interrupts]])：

- **自愿**：触发软件中断 / 执行特殊指令向操作系统索取资源。
- **被动**：来自硬件部件的信号，如**定时器**(进程占用 CPU 太久)或 IO 设备。

两种情况效果相同：CPU **先执行完当前指令**(避免不一致)，然后**程序计数器(PC)立即被覆写**，
跳到一个固定内存位置——那里存着处理中断的代码(含捕获被中断进程状态的代码)。

## 为什么纯软件解决不了

注意：**在取那些处理指令之前，PC 就已经被中断覆写了**——我们已经**不知道进程是在哪里被中断
的**了。试着用代码保存状态会遇到一连串麻烦：

1. 捕获通用寄存器——与其写进专门保留的内存位置(繁琐)，不如**压栈**更省事。
2. 标志位(flags)也重要，一起压栈。
3. 栈指针(SP)本身也要存——但**此刻 SP 指向的是用户程序的栈**！而这些代码本应由操作系统执行，
   **操作系统也是软件，有自己的栈**。在用户栈上压值，就**破坏了被中断进程的栈**。
4. 那先改 SP 指向内核栈再压？可这样**又在保存 SP 之前覆写了 SP**。再绕：先把 SP 的值存到某内存
   位置、改完 SP 后再从那里把旧值压栈……

这种细节可以没完没了地修补。但**根本死结是：PC 被中断本身覆写了**——没有任何指令能在最开头
抢先保存 PC，因为等 CPU 执行第一条指令时，那个值早已消失。这是个**罕见而有趣**的情形：需要
**硬件来支持软件**，而非反过来。

## 硬件支持

任何支持中断的架构都应提供机制，防止丢失被中断进程的状态——至少要有**最低限度的硬件支持来
保住被中断进程的 PC**。下面是两大类通用思路(每种架构各有实现)。

### 方案一：多寄存器组(multiple register sets)

最易理解：CPU 不止一组寄存器，可把不同寄存器组分给不同进程；进程被中断时，下一个进程**换用
另一组寄存器**，从而不动前一个进程的状态。

但进程可能比寄存器组多，而寄存器是硬件、运行时无法增加。不过**并不需要无限多组——两组就够**，
只要 CPU 支持操作模式：

- **普通寄存器组(common)**：供多个用户程序共享。
- **特权寄存器组(privileged)**：仅操作系统在内核态可访问(内核态几乎无所不能)。

中断发生时，CPU 切到内核态、**停用普通组、启用特权组**——于是中断**不会覆写普通组上的 PC**，
被中断进程的状态保持原样。随后：

1. 用**特权指令**读取被停用(普通)组里的寄存器值，复制进当前启用的(特权)寄存器；每条这种指令之后，
   用一条普通指令把刚取到的值写入内存(可以先到栈，但**最终会存入被中断进程的 PCB**)。其间还有
   未展示的额外指令去**查找该进程 PCB 在内存的位置**。
2. 接着读取**另一个进程的 PCB**，做相反操作：普通指令把值从内存载入当前(特权)寄存器组，再用
   特权指令转入被停用的(普通)寄存器组。
3. 若采用抢占式调度，用一条**特权指令设置定时器**；最后执行一条**中断返回特权指令**(源视频称
   "cisr"，功能类似 `iret`)，它做两件事：**切换活动寄存器组**(把用户程序锁定到普通组)、并**切回
   用户态**。下个周期 CPU 就在普通组上恢复执行**被选中的进程**。

**但此方案在现代架构中并不流行**。作者(自称非硬件专家)的猜测：在 CPU die 上复制一整套相同电路
不划算，尤其现代架构有**数百个寄存器**乃至 **SIMD 寄存器**等更复杂部件。而且其实**不必复制整组**——
只为**关键寄存器(SP、PC)** 单独留一份就够了。

### 方案二：用硬件自动化一部分

不复制寄存器组，而是把某些步骤**硬连线(hardwired)进 CPU**，像内建指令一样在**中断真正生效前**
自动执行(类比：把数据库 URL、私钥等直接硬编码进源码而非从环境读取)。两种做法：

- **硬件自动保存关键寄存器**：让**操作系统的栈指针始终对 CPU 可用**(经一个硬连线的内存位置或
  专用寄存器)。中断发生时，在 PC 被覆写去跳转处理程序**之前**，硬件**自动把 PC、SP 等关键寄存器
  压入操作系统栈**。此后由软件负责保存其余寄存器(同样要小心：在把寄存器内容存入内存前不得写它们)——
  这正是本视频前半段尝试做的事，只不过现在**被中断进程的 PC 与 SP 已由硬件保住**。
- **几乎全程由硬件完成切换**：用一个**段寄存器**指向操作系统管理的一块内存，那里有个特殊数据结构。
  某些架构里它叫**任务状态段(Task State Segment, TSS)**，对应寄存器叫**任务寄存器(task register)**。
  中断一发生，CPU **自动把所有寄存器内容整体复制进 TSS**。

也就是说，从软件视角看，上下文切换**真的可以"自动发生"**。但出于某些原因，**主流操作系统反而
避免使用**这个特性(实际比这里讲的更复杂)。

## 小结

- **上下文切换**=保存被中断进程的寄存器状态、载入下一个进程的状态；由操作系统执行，不会自动发生。
- 纯软件做不到的根因：**中断会在任何处理指令执行前覆写 PC**，没有指令能抢先保存它 → 需要硬件支持。
- 两类硬件支持：①**多寄存器组**(普通组/特权组，随模式切换)；②**硬件自动化**——自动压入 PC/SP，
  或用 **TSS** 几乎全程硬件完成切换。
- TSS 这类全硬件方案存在，但主流操作系统大多不用。
