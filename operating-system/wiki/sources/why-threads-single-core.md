## 引言

本文讲线程(threads)的基础，以及它们如何利用**并发(concurrency)** 来优化计算机资源利用率。
要点是：只要理解四个概念，线程其实很简单。**本文所讲的一切，即使在单核处理器上也成立**——
线程在单核上的意义不是"同时跑"，而是"填满 CPU 空闲的缝隙"。

## 并发与它的两个目的

在多进程系统里，若只有一个 CPU 和多个运行中的进程，操作系统会**快速地在它们之间轮换 CPU
访问权**，快到让用户产生"所有进程同时运行"的错觉。这就是**并发**，配合 **CPU 调度**，造就了
流畅的多任务体验。

但多任务只是并发**广为人知**的目的。还有一个不那么显眼的目的:**最大化计算机资源的利用**。

进程"是执行中的程序"，但这不意味着进程总在用 CPU——即使 CPU 已分配给它。例如进程在**等待
IO 资源**时无法执行指令，此时把 CPU 给它就是浪费，不如把 CPU 分给另一个**就绪**的进程。这是
并发重要的第二个原因:**填补一个进程用不上 CPU 时留下的小空隙，把它交给能用的进程**。

## 进程内为何也需要并发

此前我们把进程当作最基本的执行单位，因此**同一进程内**的可执行实体(例如两个函数)无法并发——
原因在于:每个进程只有**一个程序计数器(program counter)**，指向被中断处的指令；单个 PC 无法
同时记录两个函数各自被中断的位置，只能让一个函数等另一个跑完。

那为何不把要并发的代码拆成不同进程、用 [[concepts/interprocess-communication]] 协调?可以，
但 IPC 并不总是直观；当多段代码服务于同一个主目标、彼此高度相关时，硬拆成不同进程会"感觉不对"。

### 服务器例子

设一个服务器在 **3000 端口**监听，唯一职责是返回每个客户端的头像图片。处理一个请求要:
**accept(接受) → process(从磁盘查找并读取图片，昂贵的 IO) → respond(把图片发回)**。三步顺序
执行。单客户端没问题；但若五个请求几乎同时到达，服务器快速 accept 第一个后会**花大量时间
处理它**，第二个请求必须等第一个完成才被处理……想象一千个请求，时间线上出现大片**灰色空隙**——
CPU 闲置、什么都不做的**浪费时间**，本可用来 accept 并开始处理其他请求。

**阻塞效应(blocking effect)**:当一个任务因为另一个(本应独立的)任务在运行而无法执行，就称为
阻塞。

### 旧方案:每个客户端一个进程

很长时间里的解法是:一个 **Listener 进程**负责监听；每来一个请求，它不直接处理，而是**新建
一个进程**去服务该客户端。这样接受请求的进程与服务上一个客户端的进程**并发**运行，尽量用满
CPU。但这套方法不完美:

- 每个进程都有独立地址空间，几百个客户端尚可，**扩到数千就内存低效**；
- **创建进程开销不小**，会吃掉一部分本想省下的 CPU 时间；
- 若需维护**全局状态**，所有子进程都得靠 IPC 同步，**实现复杂**。

## 解决方案:线程

进程概念太有用，不能丢弃，但可以**稍作修改**以允许单个进程内的代码并发。进程一般包含:
ID、程序计数器、寄存器组、地址空间，以及打开的文件、IO 设备等资源。既然瓶颈是"单个 PC"，
那就**不再把 PC 直接绑定到进程**，而是给进程内每个想并发运行的**内部可执行实体各分配一个 PC**——
这些内部实体就是**线程**。

于是无需新建进程就能让同进程内的代码并发:某线程等 IO 时，其他线程照常执行，**阻塞问题解决**。

### 线程共享什么、独占什么

- **共享**:整个进程的**地址空间**(代码、堆、打开的文件、IO 设备)。
- **独占**:因为线程会被中断以把 CPU 让给别的线程，必须能保存/恢复各自状态，所以每个线程要有
  自己的 **PC、寄存器组、标志位、累加器等——本质上一份自己的 CPU 状态**，其中包括**独立的
  栈指针**。
- **每个线程有自己的栈**:否则同进程内并发的两个函数共享一个栈会互相覆盖局部变量。

> 重要澄清:线程有各自的栈，**不代表**不能读写彼此的栈。地址空间是**进程**的属性而非线程的，
> 栈位于这块共享地址空间内，因此一个线程在物理上**能**访问另一个线程的栈。该不该这么做是另一
> 回事——除非你很清楚自己在做什么，一般**应避免**直接访问其他线程的栈。线程间若要通过共享
> 内存通信，**堆(Heap)** 更合适。但即便用堆也要小心:一个线程读、另一个同时写同一块内存，
> 后果可能是灾难性的——**同步**如此关键，以至于其机制直接在**硬件**层面实现(留待未来一集)。

进程的地址空间**仍然相互隔离**:同进程的兄弟线程默认共享内存，但**不同进程的线程**要共享数据
仍需 [[concepts/interprocess-communication]]。

## 实现:PCB 与 Linux 的 task

用多线程取代传统多进程，需要**重新实现进程控制块(PCB)**。最直白的做法是再用一个结构表示线程；
而最常见的做法(Linux 内核)是:**进程和线程用同一个结构表示——叫 `task`**。这漂亮地抽象掉了
进程与线程的区别:并发不必区分"轮换进程/线程/两者"，统一说成**轮换 task** 即可，也便于用**同
一个调度器**调度两者。

无论哪种做法,都要求**每个进程至少有一个线程——主线程(main thread)**。新进程创建时操作系统会
自动创建它；即便进程不用多线程，操作系统也通过调度它的主线程来分配 CPU。

- **固定 vs 动态**:多数主流操作系统采用**动态**方式——进程以单线程启动，运行时按需创建线程
  (因此 OS 须提供动态创建线程的系统调用)。原因是编译期通常无法预知运行时需要多少线程。
- **主线程标识进程**:在很多实现里，主线程标识它所属的进程；若已派生其他线程而**主线程结束**，
  则所有其他线程会**立即终止**(有多种处理办法，另一集再谈)。

## 回到服务器例子

用多线程，可为每个到来的客户端**派生一个线程**，效果类似多进程方案，但**内存消耗低得多**；
而且通常有性能提升——**创建线程的系统调用一般比创建进程快得多**。

## 线程不是什么

- **线程不是函数**。线程执行的代码在进程内存布局的 **text 段**；线程**不包含代码**，而是通过
  自己的 PC **指向**代码。因此**多个线程可以指向完全相同的可执行代码**——服务器例子正是如此:
  所有 handler 线程不各自带一份函数，而是**指向内存中的同一个函数**。
- 多线程并发访问同一段代码**是安全的**:可执行代码与常量位于 text 段和 data 段，这两个区域
  **从不被写入**;线程在运行时产生的专属信息都在**栈或堆**里。
- 这解释了为何在 C 这类低级语言中**创建线程要传一个函数指针**——其实是在传"线程将执行的代码
  起始的内存地址"。
- **线程不是异步函数**——别被动画误导。

## 两种定义

- **操作系统视角**:线程就是上一集里"进程"扮演的角色——**最基本的执行单位**。
- **开发者视角**:线程是一种机制，用来告诉操作系统"程序里某些代码片段可以并发执行"。
- 还有一种说法:线程是**轻量级进程**，更易、更快创建。
