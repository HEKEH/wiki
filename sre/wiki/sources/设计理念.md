## 设计原则

- **API 设计原则**（8 条）、**控制机制原则**（6 条）、**架构原则**（5 条）、**引导原则** → 已系统整理为 [[concepts/设计理念]]。

- 两大核心理念：**容错性 + 易扩展性**。

## 核心 API 对象（按文档列举）

- **Pod**（最小单元、多容器共享网络与文件）：见 [[entities/Pod]]。
- **工作负载控制器**：RC、RS、Deployment、Job、DaemonSet、StatefulSet 及"业务类型 → 控制器"映射 → [[concepts/工作负载控制器]]。
- **Service / kube-proxy**（服务发现 + 分布式负载均衡）：见 [[entities/Service]]。
- **Volume / PV / PVC**：见 [[concepts/Volume存储]]（本次补充了 PV/PVC）。
- **Node**：见 [[entities/Node]]。
- **Namespace**：见 [[entities/Namespace]]。
- **Federation**：见 [[concepts/集群架构]]。
- **本次未单独建页**（仅在此登记，按需再 ingest）：
  - **Secret**：保存/传递密码、密钥、凭证，避免明文写进配置文件。
  - **User Account / Service Account**：前者标识"人"（跨 namespace），后者标识"运行中的程序/Pod"（与 namespace 相关）。
  - **RBAC**：基于角色的访问控制（Role + RoleBinding），相对 ABAC 更易扩展重用。

## 数据陈旧提示 / 勘误

源文档是早期（1.3–1.5 时代）版本，多处已过时：

- **StatefulSet 早已 GA**：源文档称其"还在 Alpha"过时——自 **v1.9 起 GA**，是生产标准对象（PetSet→StatefulSet 重命名于 1.5，准确）。
- **RBAC 早已 GA 且为默认授权**：源文档"1.3 alpha"过时——自 **v1.8 GA**，ABAC 基本弃用。
- **RC 已是 legacy**：被 Deployment/ReplicaSet 取代，新项目不应再用。
- **Federation v1 已废弃**：文中描述的"Federation 自带 API Server/Controller Manager、注册成员集群"为 v1，现代多集群用 Karmada 等（见 [[concepts/集群架构]]）。
- **Secret 默认仅 base64 编码、并非加密**：需开启 etcd encryption-at-rest 才加密静态数据；"避免明文"易被误读为"安全"。
- **默认命名空间已不止 2 个**：源文档说"初始有 default 和 kube-system 两个"，现代集群默认还有 `kube-public`、`kube-node-lease`（共 4 个）。

> 不变且正确的部分：声明式/幂等、控制逻辑只依赖当前状态、容错与优雅降级、PV↔Node / PVC↔Pod 的对称关系、Pod 多容器共享网络与文件等——这些是经得起时间的设计原则。
