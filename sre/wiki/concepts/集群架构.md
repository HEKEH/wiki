# 集群架构

一个 Kubernetes 集群由三部分组成：**分布式存储 etcd、控制节点（controller）、服务节点（Node）**。整体架构源自 [[entities/Borg]]，并可从 [[concepts/分层架构]] 的视角理解。

## 三大组成

- **控制节点（控制平面）**：负责整个集群的管理——容器调度、维护资源状态、自动扩展、滚动更新等。对应核心组件 apiserver / scheduler / controller manager（见 [[entities/Kubernetes]]）。
- **服务节点（[[entities/Node]]）**：真正运行容器的主机，管理镜像与容器，并提供集群内的服务发现与负载均衡（kubelet / container runtime / kube-proxy）。
- **[[entities/etcd]] 集群**：保存整个集群的状态，是唯一的状态数据源（只有 apiserver 直接访问它）。

> 控制平面声明并维护期望状态，服务节点执行——整体遵循 [[concepts/声明式API]]。控制平面内部各组件如何分工与协作（apiserver/controller/scheduler/kubelet 的 reconcile loop），见 [[concepts/控制平面与控制循环]]。

## 推荐 Add-ons

核心组件之外，集群通常还部署一些插件式扩展（非必需但常见）：

- **CoreDNS**（旧称 kube-dns）：为集群提供 DNS 服务，是 [[entities/Service]] 域名解析的基础。
- **Ingress Controller**：为服务提供外网入口（七层路由）。
- **Dashboard**：Web GUI。
- **监控/日志**：metrics-server（旧称 Heapster，已废弃）、Fluentd + Elasticsearch 日志采集存储查询。

> 陈旧提示：源文档列出的 Heapster、kube-dns、Federation 等名称已过时，分别由 metrics-server、CoreDNS 取代 / 废弃。

## 集群联邦（Federation）

集群联邦（Federation）用于**统一管理多个 Kubernetes 集群**——典型场景是跨地域、跨云、容灾、就近服务。

> 概念厘清：**可用区（AZ）**是同一地域内物理隔离、低延迟互联的数据中心；**一个集群通常已能跨 AZ**（标准高可用做法，无需联邦）。联邦真正解决的是**跨地域/跨集群**（远距离地域因 etcd 延迟一般不在同一集群内）。
>
> 数据陈旧提示：源文档"联邦用于跨可用区"的说法偏旧。Kubernetes **Federation v1 已废弃**，KubeFed(v2) 采用有限；当下多集群管理多用 **Karmada、Cluster API、多集群 GitOps（Argo CD）、服务网格** 等——需求仍在，技术已换代。

### 同质化还是异质化

被管理的多个集群两种模式都存在，取决于"为什么要多集群"：

- **同质化（复制型）**：同一批应用复制到各集群，为高可用/容灾/就近服务——这是**经典联邦主打的场景**。即便如此也极少 100% 相同，故采用"**统一模板 + 各集群覆盖**"（副本数、地域配置、流量权重等逐集群微调）。
- **异质化（专用型）**：各集群跑不同东西，按环境（prod/staging）、团队、能力（GPU vs CPU）、边缘等划分。

此外底层本就常不一致（K8s 版本、云厂商、节点规格），多集群工具需容忍。现代工具（如 Karmada）同时支持"复制到所有集群"与"按**放置策略**投放到特定集群"，覆盖两种模式。

## 创建集群的方式

生产部署参考官方部署指南；学习/验证可用更简单的方式：

- **minikube**：创建单机版集群最简单的方法，`minikube start` 即可，配合 `kubectl cluster-info` 查看集群信息。
- **Play with Kubernetes**（play-with-k8s.com）：免费在线体验环境，用 kubeadm 创建集群（每次最长 4 小时），并能自动显示所有 NodePort 服务端口、点击即访问。
