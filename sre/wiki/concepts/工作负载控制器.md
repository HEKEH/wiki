# 工作负载控制器

[[entities/Pod]] 是运行的最小单元，但很少直接创建 Pod；而是用**工作负载控制器**按业务类型管理一组 Pod。源文档把业务分为四类，各对应一种控制器：

| 业务类型 | 控制器 | 说明 |
| --- | --- | --- |
| 长期伺服型（long-running） | **[[entities/Deployment]]** | 无状态、常驻服务（如 Web） |
| 批处理型（batch） | **[[entities/Job]]** | 有头有尾，跑完即退出 |
| 节点后台支撑型（node-daemon） | **[[entities/DaemonSet]]** | 每个（或选定）节点跑一个 Pod |
| 有状态应用型（stateful） | **[[entities/StatefulSet]]** | 有稳定身份与独立存储 |

## RC 与 RS（副本控制器）

- **ReplicationController（RC）**：最早的保证 Pod 高可用的对象——监控并维持指定数目的 Pod 副本（少则补、多则杀；即使副本数为 1 也比裸跑 Pod 可靠）。**仅适用于无状态长期伺服型**。
  > ⚠️ RC 现为 **legacy**，已被 Deployment / ReplicaSet 取代，新项目不应再用。
- **[[entities/ReplicaSet]]（RS）**：新一代 RC，支持更丰富的 [[concepts/Label与Selector]] 匹配。一般**不单独使用**，而作为 Deployment 的底层（见 [[entities/Deployment]] 的 Deployment→ReplicaSet→Pod 链）。

## Deployment（长期伺服型）

管理无状态服务，经 ReplicaSet 维持副本，并支持滚动升级/回滚（见 [[concepts/扩缩容与滚动升级]]）。滚动升级本质是"建新 RS、副本数从 0 升满，旧 RS 降到 0"的复合操作——正因这复合性，才需要比 RS 更通用的 Deployment 来描述。详见 [[entities/Deployment]]。

## Job（批处理型）

控制**有头有尾**的批处理任务：Pod 跑完成功即退出。判断标准：**这活儿有"做完"的概念吗？有 → Job；没有（要一直在）→ Deployment**。

完成标志随 `spec.completions` 策略不同：

- 单 Pod 型：一个 Pod 成功即完成；
- 定数成功型：保证 N 个任务全部成功；
- 工作队列型：按应用确认的全局成功标志完成。

**典型场景**：数据处理/ETL、数据库迁移、批量计算（视频渲染、ML 训练）、一次性运维（备份/清理/初始化）、批量生成报表或邮件、消费有限工作队列。

> ⚠️ 易错点：**不要用 Deployment 跑批处理**——Deployment 见 Pod 退出即当故障重建，会导致"跑完又被拉起"无限循环；反之也别用 Job 跑常驻服务。

并行靠 `completions`×`parallelism`；进阶能力（Indexed Job、每索引重试 `backoffLimitPerIndex`、提前成功 `successPolicy`、TTL 清理、`suspend`）见 [[entities/Job]]。

### CronJob（定时 Job）

Job 是**跑一次**；要**按时间表周期性**运行（像 Linux 的 cron）则用 **[[entities/CronJob]]**：到点按 schedule **创建一个 Job**。典型场景：每天备份、每小时同步、定时清理。关系链：**CronJob →（按 cron 表达式）创建 Job → 创建 [[entities/Pod]]**。并发处理（Allow/Forbid/Replace）见 [[entities/CronJob]]。

## DaemonSet（节点后台支撑型）

保证**每个节点**（或经 `nodeSelector` 选定的节点）上都运行一个此类 Pod。关注点在**节点**而非业务——典型用于存储、日志、监控等每节点支撑服务。忽略 unschedulable 状态、更新策略与静态 Pod 对比见 [[entities/DaemonSet]]。

## StatefulSet（有状态应用型）

用于**有状态服务**。与 RC/RS 的"牲畜（cattle）"模型相对，它管理的是"宠物（pet）"：

| | 无状态（RC/RS/Deployment） | 有状态（StatefulSet） |
| --- | --- | --- |
| Pod 名字 | 随机，可丢弃替换 | **事先确定、稳定不变** |
| 存储 | 一般不挂或挂共享存储 | **每个 Pod 挂自己独立的存储** |
| 故障替换 | 换个名字在别处重启 | 同名 Pod 重启并**挂回原存储**，延续状态 |

稳定的名字用于关联该 Pod 对应的状态/存储。适用场景：MySQL、PostgreSQL、Zookeeper、[[entities/etcd]] 等；也可作为比普通容器更可靠的"模拟虚拟机"机制（Pod 可漂移到其他节点，存储经外挂保证高可靠）。

**StatefulSet ≠ Pod**：它和 Deployment 一样是**控制器**，管理一组有身份的 Pod。三种"稳定"分别是：① **稳定网络身份**——有序名 `db-0/1/2`，配合 [[entities/Service]] 的 **headless service（`clusterIP: None`）**，每个 Pod 还有固定 DNS `db-0.<svc>.<ns>.svc.cluster.local`，重建/漂移都不变；② **稳定存储**——经 `volumeClaimTemplates` 各绑自己的 PVC，重启总挂回原盘；③ **有序部署/伸缩**。更新策略（RollingUpdate/OnDelete、Partitions 灰度）、Pod 管理策略（OrderedReady/Parallel）、headless service + volumeClaimTemplates 细节见 [[entities/StatefulSet]]。

**为何有状态应用离不开**：集群成员要互相寻址（从库配置写"主库是 `db-0`"）、重启后认领自己的数据、保持固定角色（`-0` 常作主/种子）。若名字随机且重启就变，peer 找不到彼此、Pod 可能挂错数据盘——所以必须给"宠物"固定的**名字 + 盘 + 上场顺序**。

> 历史：1.3 以 Alpha 版 **PetSet** 发布，1.5 起重命名为 **StatefulSet**。
> ⚠️ 勘误：源文档称"StatefulSet 还在 Alpha 阶段"已过时——StatefulSet 自 **1.9 起已 GA（稳定）**，是生产可用的标准对象。

## 小结

> 选控制器先看业务类型：无状态常驻 → Deployment；批处理 → Job；每节点守护 → DaemonSet；有状态 → StatefulSet。这种"一类业务一种 API 对象"正是 [[concepts/设计理念]] 第 5 条（不同语义用不同 API 对象，而非内部算法区分）的体现。
