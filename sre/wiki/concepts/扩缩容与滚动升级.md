# 扩缩容与滚动升级

[[entities/Deployment]] 通过声明式地调整副本数和镜像，实现应用的动态扩缩容与无中断升级——这是 [[concepts/声明式API]] 在应用生命周期上的直接体现。

## 扩缩容（Scale）

修改 Deployment 的副本数 `replicas` 即可动态扩展或收缩应用：

```bash
kubectl scale --replicas=3 deployment/nginx-app
```

自动扩展出的容器会**自动加入** [[entities/Service]]，回收的容器也会**自动从 Service 移除**。

## 滚动升级（Rolling Update）

逐个替换容器以实现**无中断**的服务升级。

- **旧方式（仅针对 ReplicationController）**：

  ```bash
  kubectl rolling-update frontend-v1 frontend-v2 --image=image:v2
  kubectl rolling-update frontend-v1 frontend-v2 --rollback   # 回滚
  ```

- **Deployment 方式（推荐）**：Deployment 的更新策略默认即 `RollingUpdate`，更新镜像后自动滚动升级：

  ```bash
  kubectl set image deployment/nginx-app nginx-app=nginx:1.9.1
  ```

  策略由 `maxSurge` / `maxUnavailable` 控制：

  ```yaml
  strategy:
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
    type: RollingUpdate
  ```

### 如何触发滚动升级

**只要改动 Deployment 的 `spec.template`（Pod 模板）就会触发滚动升级**——模板变 → 新 `pod-template-hash` → 新建 ReplicaSet 逐步替换 Pod。以下方式殊途同归（最终都经 apiserver 更新对象 → 控制器 reconcile）：

```bash
kubectl set image deployment/nginx-app nginx-app=nginx:1.9.1   # 命令式快捷
kubectl edit deployment/nginx-app                              # 在线编辑活动对象
kubectl apply -f deployment.yaml                               # 声明式（推荐，GitOps 用此）
kubectl patch deployment/nginx-app -p '{...}'                  # 局部补丁
```

注意区分改动位置：

| 改动 | 效果 |
| --- | --- |
| `spec.template.*`（镜像/env/资源/标签…） | ✅ 触发滚动升级 |
| `spec.replicas` | 仅扩缩容，不滚动升级 |
| `spec.selector` | ❌ 创建后不可改 |

> 改本地 YAML 文件本身不生效，须 `apply`（或 edit/patch）提交给集群——集群不监视你的磁盘文件。声明式（改 YAML + apply）优于命令式，便于追溯与回滚（见 [[entities/kubectl]]）。

## 查看与回滚

```bash
kubectl rollout status deployment/nginx-app     # 查看进度
kubectl rollout history deployment/nginx-app    # 查看历史版本
kubectl rollout undo deployment/nginx-app        # 回滚到上一版本
```

> 注意区别：`kubectl rolling-update` 只作用于 ReplicationController；Deployment 用声明式的 RollingUpdate 策略 + `rollout` 系列命令。

## 相关补充

- **另一种策略 Recreate**：先杀光旧 Pod 再建新的（有停机，但同一时刻只一个版本）——与 RollingUpdate 的取舍见 [[entities/Deployment]]。
- **暂停/恢复**（`rollout pause/resume`）、**比例扩容**、`revisionHistoryLimit`（保留多少可回滚版本，默认 10）等进阶行为见 [[entities/Deployment]]。
- 回滚能回到多早，取决于保留的旧 ReplicaSet 数（删了旧 RS 就回不去那个 revision）。
- **自动**扩缩容（按 CPU/指标自动调副本数的 HPA、自动调单 Pod 资源的 VPA）见 [[entities/Autoscaling]]——本页讲的是手动 `kubectl scale`。
