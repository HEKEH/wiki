# 设计理念

源文档总结 K8s 最核心的两个设计理念：**容错性** 与 **易扩展性**。容错性是稳定与安全的基础；易扩展性保证对变更友好、可快速迭代增加新功能。下面是支撑这两点的具体原则——它们也是分布式系统设计的通用经验。

## API 设计原则

1. **所有 API 都是声明式的**：声明式对重复操作幂等、稳定（不怕丢操作或重复执行），向用户隐藏实现细节。隐含推论——**API 对象都是"名词"**（Service、Volume 等描述期望得到的目标对象）。详见 [[concepts/声明式API]]。
2. **API 对象彼此互补且可组合**："高内聚、松耦合"，提高对象可重用性。
3. **高层 API 以操作意图为基础设计**：从业务（调度管理容器的意图）出发，而非过早从技术实现出发。
4. **低层 API 根据高层 API 的控制需要设计**：低层为被高层使用而存在，减少冗余、提高重用。
5. **避免简单封装、不要有外部 API 无法显式知道的内部隐藏机制**：例如 StatefulSet 与 ReplicaSet 本是两种 Pod 集合，就用**不同 API 对象**定义，而非用同一个对象内部用算法区分（见 [[concepts/工作负载控制器]]）。
6. **API 操作复杂度与对象数量成正比**：上限不超过 O(N)，否则系统失去水平伸缩性。
7. **API 对象状态不能依赖网络连接状态**：分布式环境网络断开是常态。
8. **避免让操作机制依赖全局状态**：分布式系统中保证全局状态同步非常困难。

> 关于第 8 条："全局状态"指**所有组件必须同时看到、且时刻一致的那份状态**。要求"全系统先同步出一致全貌才能动手"会带来全局锁/共识的协调瓶颈，网络一分区就卡死，且无法水平扩展。K8s 的做法相反：各组件**各自 watch apiserver、按当前观察到的状态独立行动、最终收敛**（见 [[concepts/控制平面与控制循环]] 的星型拓扑），唯一事实源是 etcd（经 apiserver），组件之间不互相同步全局状态。与第 7 条的区别：第 7 条针对**网络连接活性**，第 8 条针对**是否需要全局一致快照**。这也与下面"控制逻辑只依赖当前状态"互为一体。

## 控制机制设计原则

- **控制逻辑只依赖当前状态**：故障后只要重置到某稳定状态，控制逻辑就能恢复正常——这正是 [[concepts/控制平面与控制循环]] reconcile loop 的根基。
- **假设任何错误都可能发生，并做容错处理**：局部/临时错误是大概率事件。
- **避免复杂状态机，控制逻辑不依赖无法监控的内部状态**：相互影响的子系统必须能访问到对方影响控制逻辑的状态。
- **假设任何操作都可能被拒绝或错误解析**：操作级错误不应影响系统稳定性。
- **每个模块都能在出错后自动恢复**：不因连不上其他模块而自我崩溃。
- **每个模块都能在必要时优雅降级**：划清基本功能与高级功能，基本功能不依赖高级功能。

## 架构设计原则

源文档列出的 5 条架构原则，与 [[concepts/控制平面与控制循环]] 高度一致，此处仅列要点（详见该页）：

- 只有 **apiserver** 可直接访问 [[entities/etcd]]，其他组件必须经 K8s API 访问集群状态；
- 单节点故障不应影响集群状态；
- 故障恢复后组件继续执行上次最后收到的请求（应对网络分区/重启）；
- 所有组件在内存中保持所需状态，apiserver 写 etcd，其他组件经 apiserver 更新并 **监听（watch）** 变化；
- **优先使用事件监听而非轮询**。

## 引导（Bootstrapping）原则

- **Self-hosting 是目标**；减少依赖，尤其是稳态运行的依赖；通过分层管理依赖。
- **循环依赖**的处理：接受本地文件等其他输入（其他服务不可用时可手动配置引导）；状态应可恢复/可重新发现；用临时实例创建稳态所需状态、用分布式锁/文件锁协调切换（`pivoting`）；自动重启异常退出的服务。

> 引导原则解释了 [[entities/kubelet]] 用**静态 Pod**自举控制平面的做法（用本地 manifest 起 apiserver 等，绕开"apiserver 还没起谁来起它"的循环依赖）。

## 两大核心理念

> **容错性**（稳定/安全的基础）+ **易扩展性**（对变更友好、可快速迭代）。上述所有原则——声明式、只依赖当前状态、容错降级、可组合的名词式 API——都服务于这两点。
