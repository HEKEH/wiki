# 插件机制与可扩展性

Kubernetes 把自己定位为**"构建生态系统的平台"，而非封闭的 PaaS**（见 [[sources/kubernetes简介]]）。为同时满足"适配一切环境"与"核心保持稳定"这对矛盾，K8s 把会因环境而变的部分抽出**标准接口**，交给可插拔的插件实现。

## 为什么需要插件

1. **环境千差万别，不能写死**：裸机 / 各家云 / 各种网络与存储后端，硬编码无法适配所有。
2. **核心精简稳定**：apiserver、scheduler、controller manager 只管通用编排逻辑，具体实现外包给插件，降低耦合。
3. **让生态创新**：厂商/社区实现接口即可接入，无需改 K8s 源码——这就是存储、网络方案百花齐放的原因。

> 呼应源文档："用户可以编写自己的控制器和调度器，也可以通过各种插件机制扩展系统的功能。"

## 两类插件

### 1. 运行时基础设施接口（核心组件依赖）

| 接口 | 全称（英文 / 中文） | 解决什么 | 是否有官方默认实现 |
| --- | --- | --- | --- |
| **CRI** | Container Runtime Interface / 容器运行时接口 | 用哪个容器运行时 | 曾内置 Docker（dockershim，v1.24 移除）；现须自装，主流 containerd / CRI-O |
| **CNI** | Container Network Interface / 容器网络接口 | Pod 如何联网、分配 IP | **无可用默认，必须自装**（不装则 Pod 不通）；主流 Calico/Flannel/Cilium |
| **CSI** | Container Storage Interface / 容器存储接口 | 如何对接存储后端 | 无内置驱动；厂商自行提供（见 [[concepts/Volume存储]]） |

> 记忆窍门：三者都是 **C…I = Container … Interface**，中间字母即领域——**R**untime / **N**etwork / **S**torage。（源文档中 kubelet 卷管理的旧称 **CVI** 即现在标准化的 CSI。）

这些接口的统一协作模式是 **"核心组件调用、插件实现"**：接口由 K8s 定义为标准契约（多为 gRPC），具体实现交给可替换的插件。以 CRI 为例：

- **kubelet 端是客户端**：[[entities/Node]] 上的 kubelet 调用 CRI 接口下命令（拉镜像、创建 Pod 沙箱、起停容器、查状态）。
- **container runtime 端是服务端**：containerd / CRI-O 等**实现** CRI 的 ImageService 与 RuntimeService，真正执行。
- 因为接口固定，底层 runtime 可独立替换而 kubelet 代码不动（早期 Docker 需经 dockershim 适配，v1.24 移除）。详见 [[entities/Node]]。

要点：K8s **提供"规范 + 参考实现/通用配件"，但不钦定唯一赢家**。代价是装完集群还要自己选网络/存储；收益是适配一切环境、生态竞争创新。

### 2. 功能扩展类（扩展 K8s 行为）

- **设备插件（Device Plugin）**：让 Pod 使用 GPU、FPGA 等特殊硬件。
- **调度器扩展 / 自定义调度器**：改变 Pod 调度决策。
- **准入控制 Webhook**：资源创建前拦截、校验或自动修改（如注入 Sidecar）。
- **CRD + 自定义控制器（Operator）**：给 K8s 增加全新资源类型和管理逻辑。
- **认证/授权插件**：对接企业身份系统、自定义鉴权。

这类扩展都构建在**与开发者相同的 API** 之上，与 [[concepts/声明式API]] 一脉相承：定义新的期望状态（CRD），再写控制器让现实收敛到它。
