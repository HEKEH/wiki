# 控制平面与控制循环

Kubernetes 控制平面由 [[entities/etcd]]、apiserver、controller manager、scheduler 等组件构成。它们协同实现了 [[concepts/声明式API]] 的核心闭环：**用户只声明期望状态，系统通过持续的控制循环（reconcile loop）让现实向期望收敛**。本页讲清这些组件各自的职责、两两关系，以及一次操作的端到端链路。

## 各组件职责

| 组件 | 角色 | 主动性 | 干什么 |
| --- | --- | --- | --- |
| **etcd** | 唯一状态存储 | 被动 | 保存期望状态与现状（key/value） |
| **apiserver** | 唯一入口 / 数据中枢 | 被动，不决策 | 认证、授权、准入、校验，读写 etcd；**唯一直接访问 etcd 的组件** |
| **controller manager** | 状态维护者（大脑） | 主动 | 跑各类控制循环，决定"该有哪些对象"，纠正期望与现状的偏差 |
| **scheduler** | 调度器（专用控制器） | 主动 | 只做一件事：给未调度的 [[entities/Pod]] 选 [[entities/Node]] |
| **kubelet** | 节点代理 | 主动 | 在 Node 上调 CRI 真正拉起容器，并上报现状（见 [[entities/Node]]） |

## 关键原则：一切经 apiserver，watch 的是 apiserver

controller、scheduler、kubelet **都不直接访问 etcd**，而是通过 apiserver 读写；它们 **watch 的对象是 apiserver**（由 apiserver 在内部 watch/缓存 etcd 并转发变化）：

```
   controller / scheduler / kubelet
        ▲                   │
        │ watch             │ 写回
        │（apiserver 推送变更）▼（创建/修改对象）
       apiserver  ◄── 读写 / 内部 watch ──►  etcd
                （唯一直接访问 etcd 的组件）
```

- **向上箭头 = watch（读）**：apiserver 把状态变更推送给 controller/scheduler/kubelet。
- **向下箭头 = 写回（写）**：它们把新建/修改的对象提交给 apiserver。
- apiserver ◄──► etcd：双向箭头表示**数据来回**，而非双方都能发起——**只有 apiserver 主动访问 etcd**（它是 etcd 唯一的客户端，etcd 从不主动找 apiserver）。但读/watch 虽由 apiserver 发起，结果数据是 **etcd → apiserver** 返回的（写则是 apiserver → etcd），故数据双向。apiserver 正是借内部 watch 拿到 etcd 变更，再推送给上面的组件。

好处：etcd 只有一个"守门人"，认证/校验/审计集中在 apiserver；各组件只认 apiserver 这一个统一接口，彼此**解耦**——它们靠"观察同一份状态、各做自己那段"协作，而非互相发指令。

## 两类状态与 reconcile loop

etcd 中同时保存两类状态：

- **期望状态（desired）**：用户声明的，如 `replicas=5`。
- **现状（actual / status）**：实际发生的，如当前 Running 了 3 个。

控制循环的本质是不断重复：**观察 desired 与 actual 的差 → 决策 → 行动缩小差 → 行动结果改变 actual → 写回**。这就是 K8s "消除编排需要"的实现机制（见 [[concepts/容器编排]]）。etcd 的 **Watch 机制**（见 [[entities/etcd]]）是这一切的触发基础。

## controller 与 scheduler 的关系：接力，不是上下级

- **controller 决定"要有这个 Pod"**：创建出的 Pod 是"未调度"的（`spec.nodeName` 为空）。
- **scheduler 决定"它去哪台 Node"**：watch 到没有 Node 的 Pod，跑**预选（过滤）+ 优选（打分）**选机器，写回绑定。
- scheduler 本质也是一个 watch-比对-行动的控制循环，只是**专精于"放置（placement）"**而被单独拆成一个组件，便于定制/替换（见 [[concepts/插件机制与可扩展性]]）。

> controller = 通用的"维持期望状态"大军；scheduler = 被拎出来专管"选机器"的特种兵。

## 端到端示例：把副本从 3 改成 5

```
① kubectl scale ──► apiserver 校验后把 replicas=5 写入 etcd
                    （apiserver 的活到此结束，它不会自己去建 Pod）
② controller watch 到"期望5/现状3，差2"──► 经 apiserver 创建 2 个 Pod
                    （此时 Pod 的 nodeName 为空 = 未调度，容器还没跑）
③ scheduler watch 到"有 Pod 没 Node"──► 预选+优选选机器 ──► 经 apiserver 写回绑定
④ 目标 Node 的 kubelet watch 到"有我的 Pod"──► 调 CRI 拉镜像、起容器
⑤ kubelet 把"现状：已 Running"经 apiserver 写回 etcd
```

## 重要澄清：改 etcd 不是目的，真实动作在 etcd 之外

容易误以为 controller/scheduler 就是"在 etcd 里搬数据"。其实：

- 它们的核心价值是**决策逻辑**（调度算法、副本/滚动更新逻辑），写 etcd 只是**记录意图**。
- controller 在 etcd 里创建一个 Pod 对象时，**容器还没跑**——那只是一条"应该有这个 Pod"的记录。
- **真正拉镜像、起进程的实际动作由 kubelet + container runtime 在真实机器上完成**，完全发生在 etcd 之外；做完再把现状写回。

> 类比：etcd 是工程进度白板，apiserver 是唯一能改白板的文员，controller 是项目经理、scheduler 是调度员（都在白板上登记任务），而 kubelet+runtime 是**真在工地上盖楼的工人**——楼是工地上盖起来的，不是在白板上"写"出来的。
