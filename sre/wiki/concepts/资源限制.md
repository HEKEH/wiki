# 资源限制

Kubernetes 通过 **cgroups** 提供容器资源管理，可限制每个容器的 CPU 和内存使用，实现可预测的应用性能与资源隔离。

## 设置方式

命令行（针对已有 [[entities/Deployment]]）：

```bash
kubectl set resources deployment nginx-app -c=nginx --limits=cpu=500m,memory=128Mi
```

等价于在 Pod 的容器 `spec` 中设置 `resources.limits`：

```yaml
spec:
  containers:
    - image: nginx
      name: nginx
      resources:
        limits:
          cpu: "500m"      # 0.5 个 CPU 核
          memory: "128Mi"
```

## 单位说明

- **CPU**：`500m` 表示 0.5 核（`1000m` = 1 核）。例中限制最多用 50% 的 CPU。
- **内存**：`128Mi` 表示 128 MiB（mebibyte）。

内存单位两套都支持，区别在进制——**带 `i` 是 2 的幂、不带是 10 的幂**：

| 二进制（2 的幂） | 十进制（10 的幂） |
| --- | --- |
| `Ki` / `Mi` / `Gi` / `Ti` | `k` / `M` / `G` / `T`（kilo 是小写 `k`） |

例如 `1Mi = 2²⁰ = 1,048,576` 字节，`1M = 10⁶ = 1,000,000` 字节（`Mi` 比 `M` 大约 4.9%）。所以 `128Mi ≈ 134 MB`。内存通常用 `Mi`/`Gi`（内存本按 2 的幂分配，更自然）。

> 底层由 cgroups 强制执行。资源限制是 [[entities/Node]] 上调度与资源隔离的基础，也体现了源文档所说"资源隔离：可预测的应用程序性能"（见 [[concepts/容器]]）。

## requests vs limits

容器的 `resources` 有两类值，含义不同：

- **request（请求）**：**至少要保证**的量。[[entities/kube-scheduler]] 按 request 找够空间的节点（预选 `PodFitsResources`），也是 cgroup 的保底。
- **limit（上限）**：运行时**最多能用**。超了：CPU 被**限流（throttle）**，内存被 **OOMKill**。

```yaml
resources:
  requests: { cpu: "250m", memory: "64Mi" }   # 调度保底
  limits:   { cpu: "500m", memory: "128Mi" }  # 运行上限
```

## QoS 等级（BestEffort / Burstable / Guaranteed）

K8s 按"有没有设 requests/limits"**自动推导**出 Pod 的 **QoS（服务质量）等级**——它决定**节点资源紧张时谁先被牺牲**（不是手动字段）：

| QoS 等级 | 判定条件 | 资源预留 | 资源紧张时 |
| --- | --- | --- | --- |
| **Guaranteed**（最高） | 所有容器 CPU 和内存都设了 request **且 request == limit** | 完整预留 | **最后**被驱逐/杀 |
| **Burstable**（中） | 设了部分 request/limit，但不满足 Guaranteed | 保到 request | 居中 |
| **BestEffort**（最低） | **完全没设任何 request/limit** | **零预留** | **最先**被驱逐/OOM 杀 |

- 驱逐/杀的顺序：**BestEffort → Burstable → Guaranteed**（见 [[entities/kubelet]] 驱逐）。
- 这也解释了 [[entities/kube-scheduler]] 的行为：`MemoryPressure` 的节点**不再接收 BestEffort 新 Pod**（零保障、最先被杀，调上去也是白调）。
